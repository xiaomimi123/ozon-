# 货源采集技术参考（1688 + 拼多多）

> 配套《开发文档 v3》§5.3 货源匹配模块。汇总可参考的开源实现与技术路线，供 Claude Code 开发货源采集时作为起点与避坑指引。
> 说明：以下仓库均为技术参考，非直接依赖；实际接口/加密方案会随平台更新变化，需自行验证与维护。
> 版本：v1.0 ｜ 2026-07-16

---

## 0. 总体判断（先看这个）

| | 1688 | 拼多多 |
|---|---|---|
| 采集难度 | 低 | 高 |
| 现成轮子质量 | 高，图搜字段现成可用 | 有，但都强调需持续维护 |
| 核心难点 | 基本没有 | anti_content 加密参数 + 需登录 token/优质 IP + 平台策略常变 |
| 推荐路线 | 直接爬（图搜为主） | selenium + 代理截获移动端 API（不死磕纯 JS 逆向） |
| 落地顺序 | 主力货源，先做 | 后做；建议先跑通 1688 再攻坚拼多多 |

一句话：**1688 直接爬、字段现成；拼多多用"浏览器自动化截流"而非"纯逆向"，长期省心。**

## 1. 1688 采集

### 1.1 推荐参考仓库

**图搜（以图搜款，跟卖匹配货源的核心）：**

1. **Zhui-CN/1688_image_search_crawler** ⭐ 首选
   - 地址：https://github.com/Zhui-CN/1688_image_search_crawler
   - 亮点：返回字段极全，几乎为本项目量身定做，可直接映射到 `supply_candidates` 表与五维评分。
   - 返回字段示例：`offer_id`、`product_url`、`price`、`quantity_begin`(起批量)、`quantity_prices`(阶梯价)、`gmv_price`(GMV)、`repurchase_rate`(复购率，如 45.45%)、`company_name`、`credit_level/credit_text`(信用 AAA)、`reg_capital`(注册资本)、`province/city`、`position_labels`(深度验厂/7×24H响应/先采后付)、`scores`(综合/咨询/物流/退货等分项评分)。

2. **Carmenliukang/1688_crawler-image_search_products**
   - 地址：https://github.com/Carmenliukang/1688_crawler-image_search_products
   - 用途：通过 1688 PC 端网址上传图片查相似商品，另一套图搜实现，可对照参考。

**综合采购 / CLI 工具：**

3. **superjack2050/1688-cli**
   - AI-agent 友好的命令行工具，覆盖选品、商品调研、供应商评估、采购询价、订单管理。
   - 用途：可直接包装成本项目 `Ali1688Provider` 的一个实现，或参考其命令设计。

**关键词 / 详情爬虫：**

4. **sibosend/1688-crawler**：https://github.com/sibosend/1688-crawler ，1688 通用爬虫，学习参考。
5. GitHub `topics/1688` 下的 1688 Scraper API 系列：电商分析、比价、市场调研方向可参考。

### 1.2 1688 技术路线

```
Ozon主图 → 上传1688图搜(拍立淘) → 拿候选offer列表
  → 逐个拉详情(价格/阶梯价/起批量/供应商信息/评分)
  → 写入 supply_candidates(platform='ali1688')
```
- **图搜为主、关键词为辅**：跟卖靠图匹配最准；关键词检索作为补充（Ozon 俄文标题翻译成中文后搜）。
- **账号池 + 限速**：单账号 ≥6s/次 + 随机抖动，日上限可配，触发风控冷却换号（1688 风控比拼多多松，但仍需控制频率）。
- **字段直接可用**：Zhui-CN 的返回结构基本覆盖评分所需，属性一致性/供应商质量维度所需字段都在。

## 2. 拼多多采集（硬骨头）

### 2.1 核心难点

拼多多是国内电商反爬最严的之一，主要拦路虎：
- **anti_content 加密参数**：几乎每个数据请求都要携带，由复杂 JS 生成，需解密或绕过。
- **需登录态**：列表页/搜索页请求头常需登录后的 `AccessToken`，或优质 IP 直连。
- **无标准网页端**：数据主要在移动端(H5/App)，`yangkeduo.com` 移动站数据与 App 一致。
- **策略常变**：加密方案与访问策略拼多多会不定期更新，纯逆向方案易失效。

### 2.2 推荐参考仓库

**anti_content 解密路线（纯逆向，能拿到但维护成本高）：**

1. **onetwo1/pinduoduo** ⭐ 逆向参考首选
   - 地址：https://github.com/onetwo1/pinduoduo
   - 亮点：完整实现 anti_content 解密（Get_c/Get_i/Get_s/Get_u 四对象合并于 merge.js），Python+Scrapy 全站抓取分类与商品。
   - ⚠️ 作者警告：拼多多已改策略，列表页/搜索页请求头需加登录后的 `AccessToken`，或用质量较好的 IP 直接访问。

2. **Northxw/Pinduoduo**
   - 地址：https://github.com/Northxw/Pinduoduo
   - 亮点：带 anti_content.js。作者点明三个未知参数：`flip`、`list_id`(可从网页源码正则匹配)、`anti_content`(每次请求携带，需解密)。

**selenium + 代理截获路线（推荐，更稳）：**

3. **SZFsir/pddSpider** ⭐ 推荐主路线
   - 地址：https://github.com/SZFsir/pddSpider
   - 亮点：作者明确指出无法破解 URL 里的 anticontent 导致无法重放 URL，于是改用 **selenium 结合代理，从代理中截获返回 API 的商品信息**，已爬到 90万+ 商品。
   - 意义：**不与加密算法死磕**，靠浏览器自动化真实触发请求 + 代理截流拿数据，拼多多改版也不易崩，长期维护成本更低。

4. **YangsCalvin/pinduoduo**：https://github.com/YangsCalvin/pinduoduo ，iOS 客户端逆向抓取，模拟人为操作防异常（另一种思路，成本更高，了解即可）。

5. **OFZFZS/scrapy-pinduoduo**：抓热销商品和评论，数据源为移动端 yangkeduo.com。

### 2.3 拼多多技术路线（建议）

```
【推荐】selenium/playwright + 代理 截获移动端API
  Ozon主图关键词(译中) → 浏览器打开拼多多移动端搜索
    → 真实触发请求，代理层截获返回JSON(商品/价格/销量/店铺)
    → 解析写入 supply_candidates(platform='pinduoduo')
```
- **优先"截流"而非"重放"**：让浏览器真实生成 anti_content 并发请求，你只在代理层拿返回结果，避免自己复现加密。
- **图搜能力**：拼多多以图搜款签名更复杂；若图搜难度过高，一期可先用**关键词检索**（Ozon 标题译中后搜），图搜作为后续增强。
- **独立账号池 + 优质代理**：拼多多对 IP 质量敏感，代理成本高于 1688，需单列。
- **降级**：拼多多这条腿不稳定时，任务可仅用 1688 货源完成，不阻塞主流程（SourceProvider 抽象已支持按平台启停）。

### 2.4 拼多多的务实提醒

- **别一上来啃纯 JS 逆向**：anti_content 解密方案会随拼多多更新失效，维护是长期负担。优先 SZFsir 的 selenium+代理截获路线。
- **取数成本要客户承担**：拼多多的优质代理 IP、可能的第三方兜底，都属运行成本，应在合同/说明书中列为甲方承担项。
- **可先做关键词、再补图搜**：降低一期落地风险。

## 3. 统一到 SourceProvider 抽象

两个平台都收敛到同一接口（见开发文档 §5.3），上层无感、可按平台启停与切换：

```python
class SourceProvider:
    platform: str                                   # 'ali1688' | 'pinduoduo'
    def image_search(self, image_url) -> list[Candidate]: ...
    def keyword_search(self, kw) -> list[Candidate]: ...
    def fetch_detail(self, offer_id) -> CandidateDetail: ...

# 实现：
# Ali1688Provider    —— 图搜为主(参考 Zhui-CN 字段结构)
# PinduoduoProvider  —— selenium+代理截获(参考 SZFsir)，一期可先关键词
```
- 每个 provider 独立账号池、独立限速参数、独立代理配置。
- matcher 按 `task.source_platforms` 遍历启用的 provider，结果合并去重后统一评分。
- 任一平台故障或被风控，不阻塞另一平台与主流程。

## 4. 落地顺序建议

1. **先 1688 图搜跑通**：用 Zhui-CN 的字段结构打通"图搜→候选→详情→入库→评分"，这是主力货源。
2. **拼多多先关键词、走 selenium 截流**：参考 SZFsir，先拿到基础商品数据，图搜作为增强。
3. **两平台合并评分与去重**：跨平台按图片指纹去重，同一套五维评分排序。
4. **各自独立账号池/代理/限速**，故障互不影响。

> 免责：以上仓库仅为技术参考，采集第三方平台需遵守相应平台规则与适用法律；接口与加密方案会变化，需持续验证维护。
