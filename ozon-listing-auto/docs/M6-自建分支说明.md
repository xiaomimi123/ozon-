# M6 使用说明：自建分支（改图 → 类目映射 → 定价 → 自建建品）

本说明面向已完成 M1 采集、M2 货源匹配、M3 评分审核、M4 定价与草稿确认、M5 节奏调度的用户，聚焦 M6 新增的、**只属于自建（`listing_mode=create`）分支**的三步：改图、类目属性映射、自建建品；并说明确认闸门与 live 后置项。

## 0. 目的：跟卖分支不需要改图/建类目，自建分支需要

跟卖（`listing_mode=follow`）分支挂靠的是 Ozon 已有商品的 SKU，图/类目/属性都沿用 Ozon 原商品，M4/M5 直接定价 + 挂靠即可。自建（`listing_mode=create`）分支要在 Ozon 上**新建一个商品**，没有现成的图和类目可用——候选商品的源图通常带货源水印/背景杂乱/尺寸不规范，直接传给 Ozon 会被拒或影响转化；Ozon 建品也必须挂在一个具体类目下并填写该类目要求的属性。M6 补的就是这两步「让候选商品的图和类目信息达到可建品的标准」，加上把两者接起来的自建草稿生成、确认闸门与建品调用。

## 1. 完整流程

```
采集(M1) → 匹配(M2) → 评分(M3) → 审核采用(M3, status=adopted/auto_adopted)
  → 改图(M6: ImageStudio/POST /images/process → product_images, 人工采用/弃用)
    → 生成自建草稿(M6: POST /listing/build, listing_mode=create 时分派到 build_create_drafts
                    → 译标题 + 定价(M4) + 类目建议(记忆→LLM→兜底) + 拉取已采用图 → listing_drafts mode=create)
      → 类目/图确认(M6: ListingReview 自建区 → POST /listing/{id}/confirm-category → 类目/图闸门)
        → 确认上架(M4: POST /listing/{id}/confirm, create 分支需先过闸门)
          → 配节奏 + 排期 + tick(M5, 与跟卖分支共用)
            → 逐一建品(M6: tick_publish/run_publish 按 draft.mode 分派到 create_product)
              → published + 回写 ozon_result.ozon_product_id (mock: "OZC-{offer_id}")
```

与 M4/M5 一致，M6 的验收标准也是「`docker compose up`（`IMAGE_PROVIDER=mock`/`local` 皆可、`OZON_SELLER_PROVIDER=mock`）一键跑通」，无需任何真实 Ozon/AI 生图凭据即可走完整条链路。

## 2. 改图：provider 抽象与流水线

### 2.1 为什么要抽象出 provider

改图需求分两类，处理方式完全不同：

- **本地确定性处理**（去背景、换白底、去水印、裁剪归一）：本地跑 Pillow（`rmbg` 用 `rembg`）就能出结果，不需要外部服务，速度快、可复现、无成本。
- **生成式改图**（`gen`，如根据描述重新生成商品图）：需要调用外部 AI 生图服务，涉及网络请求、API key、计费、不确定性输出。

把两者放进同一个 `ImageProvider` 接口（`app/services/imagegen/base.py`）会让本地处理绑上不必要的网络依赖。M6 的做法是按 op 分组分派（`app/services/imagegen/factory.py::process_op`）：

```python
LOCAL_OPS = {"rmbg", "whitebg", "watermark", "crop_norm"}   # 恒走 LocalProvider，不看 gen_provider
GEN_OPS = {"gen"}                                           # 按 gen_provider 参数选 mock/local/openai_compat/http
```

即：无论 `IMAGE_PROVIDER`/`gen_provider` 配的是什么，四个本地类 op 永远用本地 Pillow 处理；只有 `gen` 才会去看这个开关。这样「本地处理默认可靠」和「外部生图可插拔」两件事互不干扰。

### 2.2 各 provider

| provider | 用途 | 状态 |
| --- | --- | --- |
| `LocalProvider`（`local.py`） | 本地类 op 的唯一实现；`whitebg`/`crop_norm`/`watermark` 用纯 Pillow 真实处理；`rmbg` 惰性 `import rembg`，未装则捕获异常降级为白底并在 `meta.degraded=true` 里标记 | 已实现，whitebg/crop_norm/watermark 生产可用；rmbg 依赖 `rembg`（见 2.3） |
| `MockImageProvider`（`mock.py`） | `gen` op 的默认 provider：不发任何请求，按输入内容 hash 出一个确定性假 URL（`mock_{op}_{hash}.png`），用于本地/CI/演示 | 已实现（占位） |
| `OpenAICompatImageProvider`（`openai_compat.py`） | 走 OpenAI 兼容图像生成接口（如通义万相），供 `gen` 使用 | **占位**，`process()` 直接 `raise NotImplementedError`，live 后置 |
| `HttpImageProvider`（`http.py`） | 通用 HTTP 适配器，供字段格式不是 OpenAI 兼容的第三方生图服务（如 GRSAI/云舞 AI）使用 | **占位**，同上 |

产物统一写入 `static_dir`（默认 `server/static/images/`），返回相对 URL `/static/images/xxx.png`，经 FastAPI `StaticFiles` 挂载对外提供。

### 2.3 `rmbg` 降级与 `INSTALL_ML`

`rembg`（连同它依赖的 `onnxruntime`）体积不小，被归进 `pyproject.toml` 的 `[ml]` 可选依赖组——与 M2 的 `torch`/`cn-clip` 同一组，同样受 `INSTALL_ML` 构建参数控制。**默认镜像不装 `[ml]`**，此时 `LocalProvider._rmbg()` 的 `import rembg` 会失败，代码捕获该异常自动退化为「换白底」处理，`meta.degraded=true`，**不会报错中断流水线**。需要真实去背景效果时，把 worker（及/或需要跑同步改图的 `api`，见 2.5）镜像以 `INSTALL_ML=true` 重新构建即可，无需改代码。

### 2.4 流水线与 API

`app/workers/imager.py::run_image_process_core(session_factory, task_id, *, ops=None, static_dir=None, gen_provider="mock", fetch=None)`：

1. 查该任务下 `status in (adopted, auto_adopted)` 且有 `image_url` 的候选；
2. 对每个候选 × 每个 op（默认 `ops=["whitebg", "crop_norm"]`）建一行 `product_images`（`status=processing`），下载源图 → `process_op()` → 成功写 `result_url`/`provider`/`meta`、`status=done`，异常则该行单独标 `status=failed` + `error`，**不影响同批其余图片**（与 M4/M5 的单条独立 try/except 隔离风格一致）。
3. `sort` 字段全局递增，保证同一候选下多个 op 产物的展示顺序稳定。

API（`/images`，operator 起，采用/弃用需 reviewer）：

- `POST /images/process?task_id=&sync=`：`sync=true` 请求内同步跑完（用于本地/演示/测试立即看到结果）；`sync=false`（默认）经 ARQ 入队 `run_image_process` 异步执行——与 `/listing/publish?sync=`、`/publish/tick?sync=` 同一套 `sync` 语义。
- `GET /images?task_id=&status=`：按任务/状态查列表。
- `POST /images/{id}/approve` / `POST /images/{id}/reject`：人工逐张标记 `approved`/`rejected`。**只有 `approved` 的产物会被 `build_create_drafts` 采纳进草稿的 `images` 字段**（见第 4 节），未采用或被拒绝的图不会流入 Ozon 建品请求。

前端 **ImageStudio**（`/image-studio`）：输入任务 ID → 「开始改图」触发 `sync=true` 处理 → 产物网格逐张展示（op 标签、状态 Tag、来源候选 ID）→ 「采用」「弃用」按钮。

### 2.5 一个已知的配置留白：`IMAGE_PROVIDER` 与 `/settings/imagegen` 尚未接入调用链

`app/core/config.py::Settings.image_provider`（环境变量 `IMAGE_PROVIDER`）和 `GET/PUT /settings/imagegen`（DB 存储、admin 可配、见第 5 节）目前都**只是声明了配置项和读写接口**，还没有被 `POST /images/process`（`api/images.py`）或 ARQ 入口 `run_image_process`（`workers/imager.py`）读取——两者调用 `run_image_process_core` 时都没传 `gen_provider`，因此始终落到函数签名的默认值 `"mock"`。再加上默认流水线 `ops` 本就不含 `gen`（只有 `whitebg`/`crop_norm`），当前跑通的默认路径完全不经过 `gen`/`IMAGE_PROVIDER` 这条支线。这是刻意留出的配置面——provider 分派逻辑（2.1 节）已经就绪、也有独立单测覆盖（`test_imagegen.py`），只是「从配置读到调用参数」这一段接线留给后续把 `gen` op 接入主流程（对接真实外部生图适配器）时一并做，避免在适配器还是占位实现（`NotImplementedError`）阶段就抢先接一条目前打不通的链路。

## 3. 类目属性映射：记忆表 → LLM → 兜底

Ozon 建品要求商品挂在一个具体类目下并按该类目的属性模板填写属性。M6 用三级策略解决「新候选商品该挂哪个类目」的问题（`app/services/category_map.py::suggest_category`，`POST /category/suggest`）：

1. **记忆表命中**（`source="memory"`）：把候选标题归一化成 `signature`（去空白/转小写，截前 120 字符）去查 `category_maps` 表，命中且该记录 `confirmed=true` 就直接复用其 `ozon_category_id`/`path`/`attributes`，并把 `usage_count` 加一。同类商品（标题前缀相同）跨任务复用，不用每次都问 LLM。
2. **LLM 建议**（`source="llm"`）：未命中记忆时，把候选标题 + `CategoryTreeProvider.all_leaves()` 枚举出的叶子类目列表拼成 prompt，交给 `LLMProvider.extract_json()` 要求返回 `{"category_id": int, "path": str, "attributes": object}`；`MockLLM`（默认）的 `extract_json` 恒返回空字典，因此本地/CI 环境下这一级总是判定「无效」而落到第 3 级——这是预期行为，真实 LLM（`LLM_PROVIDER=openai`）接上后才会真正走到这一级。
3. **兜底默认**（`source="fallback"`）：LLM 未给出有效 `category_id` 时，兜底成固定类目 `category_id=15621048`（mock 类目树里的顶层节点「Дом」），确保 `suggest_category` 永远有返回值、不会因为分类失败而卡住整条流水线。

**记忆表复用闭环**：人工在确认页调整/确认某条草稿的类目属性时（`POST /listing/{draft_id}/confirm-category` → `confirm_category()`），除了把 `category_id`/`attributes` 写回该草稿，还会按草稿标题的 `signature` upsert 一条 `category_maps` 记录（`confirmed=true`）。下次遇到标题前缀相同的候选，`suggest_category` 就能直接命中记忆表第 1 级——人工确认得越多，后续 LLM 调用越少，这是这套三级策略要达到的「越用越准、成本递减」效果。

**类目树来源**（`app/services/category_tree.py`）：`mock`（默认）是写死的 6 节点小树（3 个一级分类 `Одежда`/`Обувь`/`Дом` + 3 个叶子类目），供本地/CI/演示用，也是前端 `GET /categories` 下拉的数据源；`real`（走 composer-api `categoryChildV3` 获取真实 Ozon 类目树）**尚未实现**，`get_category_tree("real")` 目前只是 import 一个不存在的模块占位，live 后置。

## 4. 自建草稿生成与确认闸门

### 4.1 `build_create_drafts`：把改图 + 类目结果拼进草稿

`POST /listing/build` 会先查该任务的 `listing_mode`：`create` 分派到 `app/services/listing_builder.py::build_create_drafts`，`follow` 分派到 M4 的 `build_follow_drafts`（两者共用同一个入口 API）。`build_create_drafts` 对每个已采用候选：

1. `llm.translate(candidate.title, target_lang="ru")` 译出俄文标题（`MockLLM` 恒等透传，`title`/`description` 目前用同一段文本）；
2. 复用 M4 `pricing.py::price_candidate` 定价，逻辑与跟卖分支完全一致（含最低价拦截、`below_min` 状态）；
3. `suggest_category()`（第 3 节的三级策略）拿到类目建议；
4. 查该候选下 `status="approved"` 的 `product_images.result_url` 列表（按 `sort` 排序）作为草稿的 `images`；**没有已采用图时 `images=[]`**，等人工在确认页补齐。
5. 写入一条 `listing_drafts`（`mode="create"`）。

按 `(task_id, candidate_id)` 幂等——同一候选重复调用 `/listing/build` 不会生成第二条草稿（`skipped` 计数），可以多批次调用（先给部分候选改完图/确认类目，再补建其余）。

### 4.2 确认闸门：没类目/没图不让确认

`app/workers/publisher.py::confirm_draft`：

```python
if d.mode == "create" and (d.category_id is None or not d.images):
    return {"draft_id": draft_id, "status": d.status, "error": "自建草稿需先确认类目与图片再确认上架"}
```

即 `mode="create"` 的草稿在 `POST /listing/{id}/confirm` 时，只要 `category_id` 为空**或** `images` 为空列表/`null`，请求会直接带 `error` 字段返回、草稿状态不变（不会变成 `confirmed`），后续也就进不了 M5 排期/建品。这是刻意设的闸门：自建商品没有类目和图就去建品，Ozon 侧必然拒绝或产出低质量商品，不如提前挡在确认这一步。**跟卖分支（`mode="follow"`）不受此闸门影响**，沿用 M4 原有行为。

对应的操作顺序建议：先在 ImageStudio 改图 + 采用满意的图 → 在 ListingReview 的自建区调 `POST /category/suggest` 拿建议（或手动改）→ `POST /listing/{id}/confirm-category` 确认类目属性（此时如果 `build` 时还没图，需要在这之前先把图跑完并采用，因为 `images` 是 `build_create_drafts` 一次性写入的，`confirm-category` 不会重新去查图；若中途才补了图，需重新触发 `/listing/build` 让同一候选走 `skipped` 分支……实际上 `skipped` 意味着不会更新——**若图片是在 `build` 之后才被采用的，需要人工直接编辑该草稿的 `images` 字段或走一次数据修复，这是当前实现的一个操作顺序假设：改图应先于 `build`**）→ 再 `POST /listing/{id}/confirm`。

### 4.3 自建建品：mock-first

`app/workers/publisher.py::_call_seller()` 按 `draft.mode` 分派 Ozon 写入调用：`create` → `OzonSellerProvider.create_product(...)`，`follow` → 原有的 `create_follow_offer(...)`；这一个分派函数被 `run_publish_core`（M4 直发）和 `tick_publish`（M5 节奏 tick）两处共用，因此**自建分支的建品复用了跟卖分支已有的 M4/M5 上架路径**（`/listing/publish`、`/publish/schedule` + `/publish/tick`），不需要单独一套上架 API。

- `MockOzonSeller.create_product()`（默认）：确定性返回 `ok=True`、`ozon_product_id=f"OZC-{offer_id}"`、`status="imported"`，供本地/CI/演示验证「build → confirm → publish/tick → published + 回写 ID」全链路。
- `RealOzonSeller.create_product()`：**占位实现**——构造了请求体（`offer_id`/`name`/`description_category_id`/`attributes`/`images`/`price`/`stock`/`barcode`），但**不发起真实网络请求**，恒返回 `ok=False`、`error="RealOzonSeller.create_product 未联调(live 校正)"`。真实 Ozon 建品端点（`POST /v2/product/import`）的字段名、鉴权方式、返回结构需要在 live 联调时对照官方文档校正——这一点与 M4 `RealOzonSeller.create_follow_offer()`（同样是占位）一致，是同一批 live 后置项。

`OZON_SELLER_PROVIDER` 环境变量与 M4/M5 语义一致：只影响异步路径（`/listing/publish?sync=false` → `run_publish`，`/publish/tick?sync=false` → `run_publish_tick`），`sync=true` 的同步路径恒用 `mock`。

## 5. AI 生图配置中心

`GET/PUT /settings/imagegen`（admin，`app/api/imagegen.py`）：

```
PUT /settings/imagegen
{"provider": "openai_compat", "img_base_url": "https://xxx/v1", "img_api_key": "sk-...",
 "img_model": "wanx-v1", "fallback": "local,mock"}
```

字段：`provider`（`mock`/`local`/`openai_compat`/`http`）、`img_base_url`、`img_api_key`（`FERNET_KEY` 加密存储；`GET` 返回时脱敏为 `"***"`；`PUT` 时该字段留空表示不改动已存的密钥，避免管理员只改其他字段时把密钥静默清空）、`img_model`、`fallback`（降级顺序，逗号分隔字符串，供后续多级降级用）。前端 **AI 生图配置页**（`/settings/imagegen`，`ImagegenSettings.tsx`）提供表单读写。如 2.5 节所述，这套配置目前是独立于改图调用链的配置面，尚未被 `run_image_process_core` 读取。

## 6. 数据模型（迁移 `0006_m6_create_branch`）

| 表/列 | 说明 |
| --- | --- |
| `product_images`（新表） | 改图产物：`task_id`/`candidate_id`/`source_url`/`op`/`provider`/`result_url`/`sort`/`status`（`pending`/`processing`/`done`/`failed`/`approved`/`rejected`）/`error`/`meta`（JSONB）。索引 `(task_id, status)`、`candidate_id`。 |
| `category_maps`（新表） | 类目映射记忆表：`signature`（唯一索引）/`source_hint`/`ozon_category_id`/`ozon_category_path`/`attributes`（JSONB）/`confirmed`/`usage_count`。 |
| `listing_drafts.title`/`description`/`category_id`/`attributes`（JSONB）/`images`（JSONB） | 新增列，仅 `mode="create"` 的草稿会写入非空值。 |
| `listing_drafts.ozon_product_id` | 由 `NOT NULL` 改为可空——自建草稿在建品成功前没有 Ozon 商品 ID（跟卖草稿沿用原有的挂靠目标 SKU 语义，不受影响）。 |

## 7. 改图产物存储：`server/static/images/` 与 Docker 共享卷

产物落盘到 `server/static/images/`（`os.makedirs` 运行时创建，已加入 `.gitignore`，不进版本库），FastAPI 在 `main.py` 里用 `StaticFiles(directory="static")` 挂载到 `/static` 路径对外提供访问。`POST /images/process?sync=true` 在 `api` 容器内同步跑（写产物到 `api` 容器的 `static/`），`sync=false` 由 `worker` 容器异步跑（写到 `worker` 容器的 `static/`）——两个容器如果不共享同一份磁盘，`api` 侧的 `GET /images` 查出的 `result_url` 在 `worker` 写的产物场景下会指向一个 `api` 容器本地不存在的文件，`/static/xxx.png` 会 404。`docker-compose.yml` 因此给 `api`/`worker` 两个服务挂了同一个宿主机目录卷 `./server/static:/app/static`，保证两个容器读写的是同一份产物，且容器重建/`docker compose down`（不加 `-v`）不丢产物。

**Live 后置项**：这个共享卷方案只解决了「`api`/`worker` 之间可见」，还没解决「Ozon 服务端可见」——真实建品时 `images` 字段需要传给 Ozon 一个**公网可达**的图片 URL（或走 Ozon 的图片上传接口），当前 `/static/...` 只在本机/内网可访问。生产环境需要把这个静态目录换成对象存储 + CDN 直连（或给 `api`/`worker` 所在环境配一个公网可达的反代域名），这一步与 `RealOzonSeller.create_product` 的 live 联调是配套的，留到 M7（公网部署）或 live 接入阶段一起做。

| `rmbg` 真实去背景 | 默认镜像无 `rembg`，自动降级白底 | worker（及需要同步改图的 `api`）镜像以 `INSTALL_ML=true` 构建 |
| 改图产物公网可达 | `/static/...` 仅本机/内网可访问 | 生产环境换对象存储 + CDN，或给静态目录配公网反代（见第 7 节） |

## 常见问题

- **`POST /listing/{id}/confirm` 对自建草稿返回 `error: 自建草稿需先确认类目与图片再确认上架`？** 检查该草稿 `category_id` 是否已设置、`images` 是否非空——先在 ImageStudio 采用至少一张图，并通过 `/listing/{id}/confirm-category` 确认类目，再确认上架。
- **改完图后 `build_create_drafts` 的 `images` 还是空？** `build_create_drafts` 只在生成草稿那一刻查一次 `approved` 图；如果改图/采用图发生在 `/listing/build` 之后，该候选已存在草稿会被 `skipped`（幂等）而不会补上新采用的图，需要人工处理（见 4.2 节末尾）。建议顺序：先跑完改图并采用满意的图，再调 `/listing/build`。
- **`POST /category/suggest` 一直返回 `source="fallback"`？** 本地/CI 默认 `LLM_PROVIDER=mock`，`extract_json` 恒返回空，所以第 2 级必然失效、总是落到兜底——这是预期行为；接上真实 `LLM_PROVIDER=openai` 后才会看到 `source="llm"`。
- **想验证记忆表复用？** 对同一批标题相近的候选先跑一次 `confirm-category` 确认类目，再对新的同类候选调 `POST /category/suggest`，应能看到 `source="memory"` 且响应里的 `category_id`/`attributes` 与之前确认的一致。
- **`GET /images` 返回的 `result_url` 打不开（404）？** 确认是用 `docker compose up` 启动且 `api`/`worker` 都挂了 `server/static` 共享卷（本地非 Docker 开发时两者本来就跑在同一台机器、同一份 `server/static/`，不存在这个问题）。
