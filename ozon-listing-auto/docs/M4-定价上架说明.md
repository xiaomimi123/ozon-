# M4 使用说明：跟卖定价与挂靠上架（配店铺 → 配定价 → 生成草稿 → 确认 → 挂靠）

本说明面向已完成 M1 采集、M2 货源匹配、M3 评分审核的用户，走一遍 M4 跟卖定价与挂靠上架的完整操作流程，并说明如何尝试切换到真实 Ozon 写入。

## 0. 前置说明：mock 优先，M4 是「直接挂靠」，不含节奏调度

- 与 M2/M3 一致，M4 的验收标准也是「`docker compose up`（mock Ozon 写入）一键跑通」。默认 `MockOzonSeller`：确定性返回挂靠成功，无需任何真实 Ozon 店铺凭据即可跑通「草稿 → 确认 → 挂靠」全链路，用于开发/CI。
- **M4 范围是「草稿确认后立即挂靠」**，不含上架时间/节奏调度（错峰、限速、定时批量等）——这些留给 **M5（上架节奏调度）**。M4 的 `POST /listing/publish` 一旦调用，会把所有 `confirmed` 状态的草稿在本次请求/任务内逐条挂靠完，没有排队等待。

## 1. 完整流程

```
采集(M1) → 匹配(M2) → 评分(M3) → 审核采用(M3)
  → 建店铺(M4: POST /shops, api_key 加密存储)
  → 配定价参数(M4: PUT /settings/pricing)
  → 生成草稿(M4: POST /listing/build, 按已采用候选+定价参数算进价/售价/毛利率)
    → 草稿确认(M4: 人工 POST /listing/{id}/confirm 或按开关 POST /listing/auto-confirm)
      → 挂靠上架(M4: POST /listing/publish, mock 挂靠→回写 Ozon 商品ID)
        → (可选)切真实 Ozon 写入, 见第 5 节
```

## 2. 建店铺

前端「店铺管理」（`/shops`）或直接调 API（admin 权限）：

```
POST /shops
{"name": "我的Ozon店铺", "client_id": "<Client-Id>", "api_key": "<Api-Key>", "is_sandbox": true}
```

`api_key` 用 `FERNET_KEY` 加密后存入 `shops.api_key_encrypted`，响应（以及 `GET /shops` 列表）均**不回传** `api_key` 字段，做到凭据不落地明文、不回显。`is_sandbox` 默认 `true`，仅作标记字段，当前不影响写入行为（走哪个 provider 由挂靠时传入的 seller 决定，见第 5 节）。

草稿关联店铺后，挂靠时会按 `shop_id` 取该店铺的 `client_id` + 解密后的 `api_key` 去调用 `OzonSellerProvider`；若草稿未关联店铺（`shop_id=null`），挂靠时 `client_id`/`api_key` 按空字符串处理（`MockOzonSeller` 不校验，`RealOzonSeller` 实际调用会因鉴权失败而返回 `failed`）。

## 3. 配定价参数

定价参数按 `PUT /settings/pricing`（admin 权限）以 category=`pricing` 存入配置中心（同 M1 配置中心机制，逐 key 加密存储）：

```
PUT /settings/pricing
{"mode": "builtin", "commission_rate": 0.15, "fulfillment_rate": 0.10, "fx": 13.0,
 "target_margin": 0.20, "logistics": 5.0, "min_price": 0.0, "strike_coeff": 1.3}
```

不写的字段回落到默认值（`app/services/pricing.py::DEFAULT_PRICING`）。`GET /settings/pricing` 读取时脱敏（所有值显示为 `***`，因为写入时统一标记 `is_secret=True`）。**当前无前端页面维护该配置**，需通过 API 客户端（curl/Postman 等）以 admin 账号调用。

### 3.1 两种定价模式

- **`mode=builtin`（默认）——内置毛利率反推**：

  ```
  到手价 landed = 货源到手价(cost, 人民币) + logistics(运费)
  denom = 1 - target_margin - commission_rate - fulfillment_rate
  售价(RUB) = landed / denom × fx        # denom ≤ 0 时视为不可定价, 直接拦截
  ```

  即：先设定「要拿到的目标毛利率」，反推出覆盖平台佣金、履约费率、目标毛利后的售价。

- **`mode=formula`——simpleeval 自定义公式**：`formula` 字段写一个纯算术表达式，可用变量：`cost`（货源到手价）、`logistics`、`commission_rate`、`fulfillment_rate`、`fx`、`weight`（Ozon 商品重量）、`target_margin`、`min_price`。例如：

  ```json
  {"mode": "formula", "formula": "(cost + logistics + weight * 0.5) * fx * 1.3"}
  ```

  出于安全考虑，公式求值环境**禁止函数调用与属性/方法访问**（`simpleeval` 的 `functions` 置空、`ast.Attribute` 节点被剔除），只能对上述数值变量做加减乘除等纯算术运算；公式写错、引用未知变量或求值异常时，不会抛错到调用方，而是安全兜底为该候选「拦截」（`blocked=True`，草稿状态 `below_min`）。

### 3.2 最低价保护

无论哪种模式，算出的售价 `≤ 0` 或**低于配置的 `min_price`**，该候选都会被拦截，对应草稿状态为 `below_min`，不会进入确认/挂靠环节——避免亏本价或异常价被挂到 Ozon 上。

划线价 `strike` 按 `strike_coeff`（默认 1.3）从售价换算展示，仅供参考，不参与拦截判断。

## 4. 生成草稿 → 确认 → 挂靠

### 4.1 生成草稿

在已有采用候选（M3 审核台采用/自动采用，`status` 为 `adopted`/`auto_adopted`）的任务上：

```
POST /listing/build?task_id=<id>&shop_id=<shop_id, 可选>
→ {"built": <生成条数>, "blocked": <被最低价拦截条数>, "skipped": <已存在跳过条数>}
```

`build_follow_drafts()` 按任务下每个已采用候选生成一条 `listing_drafts` 记录（**幂等**：同一 `task_id`+`candidate_id` 已存在草稿则跳过，不会重复生成），写入：

- `target_ozon_sku` / `barcode`：跟卖目标 Ozon 商品的 SKU/条码
- `cost`：到手成本价（货源价 + 运费）
- `price`：定价引擎算出的售价（RUB）
- `margin`：毛利率
- `pricing_detail`：定价明细（记录用的模式 `denom_mode`、汇率 `fx` 等）
- `status`：售价合法为 `draft`（待确认），被最低价拦截为 `below_min`

前端「上架审核」页（`/listing`）：输入任务 ID、可选店铺，点「生成草稿」即调用此接口，草稿列表实时展示进价/售价/毛利率/状态。

### 4.2 草稿确认（人工闸门 / 按阈值自动确认）

草稿默认停留在 `draft` 状态，需人工确认才能进入挂靠：

```
POST /listing/{draft_id}/confirm   →  {"draft_id": .., "status": "confirmed"}
```

前端草稿列表中 `status=draft` 的行有「确认」按钮，逐条点击确认。

若任务的 `review_config.listing_review_required=false`（免人工确认），可调用：

```
POST /listing/auto-confirm?task_id=<id>
→ {"confirmed": <n>}
```

`apply_auto_confirm()` 读取任务 `review_config`：`listing_review_required` 为 `true`（默认）时直接返回 `{"confirmed": 0}`、不做任何操作；为 `false` 时，把该任务下 `status="draft"` 的草稿——若还设置了 `listing_score_min`，则只对关联候选 `score_total >= listing_score_min` 的草稿——批量置为 `confirmed`。前端「按开关自动确认」按钮即调用此接口。

### 4.3 挂靠上架

```
POST /listing/publish?task_id=<id>&sync=true|false（默认 false）
```

对该任务下所有 `confirmed` 状态的草稿逐条调用 `OzonSellerProvider.create_follow_offer()`：

- 成功：草稿状态置为 `published`，`ozon_result` 写入 `{"ozon_product_id": ..., "status": ...}`
- 失败（含店铺凭据解密失败、seller 调用异常等）：草稿状态置为 `failed`，`error` 记录异常信息；**单条草稿失败不影响同批其余草稿继续挂靠**

`sync=true`：请求内同步跑完，返回 `{"published": n, "failed": n}`，适合本地/演示/测试快速验证。
`sync=false`（默认）：经 ARQ 入队，由 `worker` 容器异步执行 `app.workers.publisher.run_publish`，返回 `{"status": "queued"}`，进度需另调 `GET /listing/monitor` 查看。

**seller provider 选择**（M5 起）：`sync=true`（同步演示路径，`app/api/listing.py`）**恒用 `get_ozon_seller("mock")`**，供本地/演示/测试跑通全链路，不对接真实 Ozon；`sync=false`（异步 worker `run_publish`）则**按 `OZON_SELLER_PROVIDER` 环境变量选择**（默认 `mock`，设为 `real` 走 `RealOzonSeller`）。切真实见第 5 节。

查看草稿状态分布：

```
GET /listing/monitor?task_id=<id>
→ {"counts": {"draft": .., "confirmed": .., "published": .., "failed": .., "below_min": ..}}
```

前端「上架审核」页的「挂靠上架」按钮即调用 `sync=true` 的同步路径，便于演示时立即看到结果。

## 5. 切换到真实 Ozon 写入（RealOzonSeller）

`app/services/ozon_seller/real.py` 提供了 `RealOzonSeller`：以店铺的 `Client-Id`/`Api-Key` 请求头，用 `httpx` 调用**真实 Ozon Seller API**（`api-seller.ozon.ru`）。**异步 worker 路径已接入 `OZON_SELLER_PROVIDER` 环境变量**（M5 起，`run_publish`/`run_publish_tick` 按 `settings.ozon_seller_provider` 选 provider）；`sync=true` 的同步演示路径仍固定用 `mock`。要在开发环境验证真实挂靠效果：

1. 在「店铺管理」（`/shops`）配置真实/沙箱的 Ozon 店铺 `client_id` + `api_key`（Ozon 卖家后台的 Client-Id / Api-Key），`is_sandbox` 按需设置。
2. 设置 `OZON_SELLER_PROVIDER=real` 并走异步路径（`POST /listing/publish?sync=false` 或节奏 tick），worker 即用 `RealOzonSeller`；同步演示路径（`sync=true`）恒为 mock，不受此环境变量影响。
3. **端点已对齐真实 Ozon Seller API**（子项目 E，不再是占位实现）：
   - 跟卖 `create_follow_offer` → `POST /v1/product/import-by-sku`（按目标 SKU 克隆卡片建 offer；`result.unmatched_sku_list` 非空视为失败；成功返回的 `task_id` 作为 `ozon_product_id`，状态 `pending_review`）。
   - 自建 `create_product` → `POST /v3/product/import`（同样返回 `task_id`）。**已知数据缺口**：真实自建审核还需要 `type_id`（类目下商品类型）+ 尺寸/重量 + 属性字典值 id，当前 `listing_drafts` 不携带这些字段，真实调用大概率会被 Ozon 审核拒绝，需后续扩展草稿字段采集或人工补齐——**不要假设自建在真实环境端到端可用**。
   - 审核轮询 `get_product_status` → `POST /v1/product/import/info`（`{"task_id": ...}`），`imported`→`approved`、`failed`→`rejected`，其余（含请求异常）→`pending`。
   - 价格转为 Ozon 期望的干净数值字符串（`2300.0`→`"2300"`）；属性 `{attr_id:value}` 转换为 Ozon `[{complex_id,id,values:[{value}]}]` 结构。
   - **库存未包含在本轮**：Ozon 库存需单独调用 `POST /v2/products/stocks`，当前实现只在 import 请求体内提交价格，库存对接留作后续。
4. 真实网络冒烟测试见 `server/tests/test_live_ozon_seller.py`（`@pytest.mark.live`，默认跳过）：

   ```bash
   cd server
   OZON_CLIENT_ID=.. OZON_API_KEY=.. OZON_TARGET_SKU=.. \
     .venv/bin/python -m pytest tests/test_live_ozon_seller.py -m live -v
   ```

   三个环境变量任一未设时自动 `skip`；对真实/沙箱目标 SKU 发起一次 `create_follow_offer`，断言拿到非空响应，`ok=True` 时继续轮询 `get_product_status` 断言落在 `approved`/`pending`/`rejected` 三态之一。

「mock/real 切换」已在 M5 做成可配置项（`OZON_SELLER_PROVIDER`，异步路径生效）。跟卖 + 审核轮询两个端点已在子项目 E 完成真实对齐；仍待后续迭代跟进的收尾工作：自建 `create_product` 的数据缺口（`type_id`/尺寸/属性字典 id）与库存 `POST /v2/products/stocks` 的对接。完整说明见 [`README.md`](../README.md)「真实 Ozon 上架说明」一节。

## 常见问题

- **`/listing/build` 返回 `built: 0`？** 确认任务下已有候选被 M3 审核台采用（`status=adopted`/`auto_adopted`），或该任务的候选已全部生成过草稿（幂等跳过，体现在 `skipped`）。
- **草稿状态一直是 `below_min`？** 说明按当前定价参数算出的售价 `≤ 0` 或低于 `min_price`：检查 `commission_rate + fulfillment_rate + target_margin` 是否已 `≥ 1`（内置模式分母非正会直接拦截），或自定义公式是否写错、`min_price` 是否设得过高。
- **`POST /listing/auto-confirm` 返回 `confirmed: 0`？** 检查任务 `review_config.listing_review_required` 是否确实为 `false`（为 `true` 时接口直接返回 0），以及候选 `score_total` 是否达到 `listing_score_min`（如果设置了该阈值）。
- **`/listing/publish` 之后草稿变成 `failed`？** 常见原因是草稿未关联店铺（`shop_id=null`）或关联的店铺凭据解密失败；查看草稿的 `error` 字段获取具体异常信息，单条草稿失败不会影响同批其余草稿。
- **想验证真实挂靠效果，配了真实店铺凭据后 `POST /listing/publish` 还是走 mock？** 如第 5 节所述，当前代码固定使用 `MockOzonSeller`，配置店铺凭据本身不会切换 provider；需要临时改动 `app/api/listing.py`/`app/workers/publisher.py` 中的 `get_ozon_seller("mock")` 调用点，且 `RealOzonSeller` 的真实端点仍待联调校正。
