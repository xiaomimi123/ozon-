# M2 使用说明：货源匹配（账号池 → 双源候选 → 跨平台去重）

本说明面向已按 README 完成 `docker compose up`（默认 `EMBEDDER=mock` + `INSTALL_ML=false`，无需 torch 即可跑通）的用户，走一遍 M2 货源匹配的完整操作流程，并说明如何切换到真实 1688 抓取与真实 CLIP 去重。

## 0. 前置说明：mock 优先，保持轻量

- M2 的验收标准是「`docker compose up`（mock embedder）一键跑通」。默认镜像 **不装** torch/cn_clip（`[ml]` 依赖组，体积达数 GB），只有显式设置 `INSTALL_ML=true` 才会安装。
- 因此默认演示路径全程用 `mock`：货源 provider 用内置 fixtures（`MockSourceProvider`），去重用 `MockEmbedder`（SHA256 确定性归一化 512 维向量，无需任何模型）。
- 只有需要真实抓取 1688 / 真实 CLIP 语义去重时，才按第 5 节切换。

## 1. 建货源账号

以 `admin` 身份调用账号池 API（前端若已提供对应页面，操作等价）：

```
POST /accounts
{
  "platform": "ali1688",       # 或 "pinduoduo"
  "label": "1688-账号A",
  "credentials": {"cookie": "..."},   # 明文传入，后端用 FERNET_KEY 加密存储
  "daily_limit": 200,
  "min_interval_sec": 6
}
```

- 响应（`AccountOut`）不含凭据字段，做到脱敏；`GET /accounts`、`PUT /accounts/{id}` 同理。
- `daily_limit` / `min_interval_sec` 控制该账号的每日调用上限与最小请求间隔，由账号池 `acquire()` 在匹配时自动限速取号。
- mock 演示路径下账号池仍会被匹配 worker 调用取号（用于限速/换号逻辑演示），但 `MockSourceProvider` 不会真正使用 cookie 发起网络请求，所以 `credentials` 填任意占位值即可。

## 2. 采集商品（复用 M1）

按 [`docs/M1-使用说明.md`](./M1-使用说明.md) 新建并启动一个采集任务（`provider=mock` 即可），采集出的 `OzonProduct` 是货源匹配的输入——匹配 worker 会按商品的 `main_image_url`（图搜）与 `title`（关键词搜）去两个货源平台找候选。

新建任务时记得在 `source_platforms` 字段中指定要匹配的货源平台，如 `["ali1688", "pinduoduo"]`。

## 3. 启动匹配

```
POST /match/start?task_id=<id>&sync=true
```

- `sync=true`：请求内同步跑完（适合本地/mock 快速演示，一次性遍历该任务下所有商品）。
- `sync=false`（默认）：经 ARQ 入队，由 `worker` 容器异步执行（`app.workers.matcher.run_match`），适合真实抓取场景（避免长请求超时）。

匹配 worker 对每个商品：

1. 遍历 `source_platforms` 中启用的平台，各自 `acquire()` 取账号 → 图搜 + 关键词搜 → 收集候选（单平台故障用 `try/except` 隔离并 `report_risk` 触发冷却换号，不阻塞其他平台/商品，见 spec §9 降级要求）。
2. 同一商品下所有平台候选合并后按 `(platform, offer_id)` 去重，再整体交给 `dedup_and_upsert` 做**一次跨平台 CLIP 聚簇去重**入库——只有合并后一次性聚簇，跨平台的同款才能被正确折叠。
3. 游标 (`match_cursor`) 记录到已处理的最后一个商品 id，暂停/失败后可从游标续跑。

查看进度：

```
GET /match/monitor?task_id=<id>
→ {"match_status": "running|done|paused|failed", "match_stats": {"products":..,"candidates":..,"platforms_skipped":..,"platforms_failed":..}}
```

暂停：`POST /match/pause?task_id=<id>`（下次 `/match/start` 会从游标续跑）。

## 4. 查看候选

```
GET /candidates?task_id=<id>&ozon_product_id=<可选>&platform=<可选 ali1688|pinduoduo>&only_representative=<可选>&page=1&page_size=20
```

返回分页列表，每条候选（`CandidateOut`）含：

| 字段 | 说明 |
| --- | --- |
| `platform` / `offer_id` | 来源平台 + 平台内商品标识 |
| `title` / `price` / `currency` / `quantity_begin` | 标题、单价、币种、起订量 |
| `image_url` / `detail_url` | 图片与详情链接 |
| `supplier_name` / `supplier_info` | 供应商名称与信用等级/复购率等详情（1688 解析器提供） |
| `dedup_group` / `is_representative` | 跨平台聚簇分组 id；同簇内只有一条 `is_representative=true`（其余为同款折叠项，仍保留在库供审计/切换） |

`only_representative=true` 时只返回每簇的代表候选，即「近似重复折叠、双平台不同款各自保留」后的最终候选清单——这是审核台（M3）后续要消费的视图。

## 5. 切换到真实抓取 / 真实 CLIP

mock 链路验证通过后，按需分别切换以下两处（互相独立，可只切一个）：

### 5.1 真实 1688 图搜

1. 在账号池为 `platform="ali1688"` 的账号写入真实可用的登录态 cookie（`credentials: {"cookie": "..."}`）。
2. 新建/编辑采集任务时把货源 provider 工厂使用的平台保持 `ali1688`（`Ali1688Provider` 已是 `httpx` + cookie 真实实现，非 mock），无需额外配置切换代码——账号池取到真实 cookie 后 provider 自动用其发起请求。
3. 对应真实网络冒烟测试 `server/tests/test_source_live.py` 默认跳过（标记 `@pytest.mark.live`），需要显式执行 `pytest -m live` 才会真实发起网络请求。
4. **拼多多目前不可切换为真实抓取**：一期只完成了 JSON 解析层 `parse_pdd_items`（价格分转元、字段映射，已有单测覆盖），`PinduoduoProvider.image_search` / `keyword_search` 均为占位实现，调用会抛 `NotImplementedError`。真实抓取需要 `selenium` + 代理截获拼多多移动端 API JSON，留待后续迭代接入；当前 `source_platforms` 中包含 `pinduoduo` 时，matcher 会捕获该异常按平台故障处理（计入 `platforms_failed`，不中断其他平台/商品）。

### 5.2 真实 CLIP 跨平台去重

1. 编辑 `.env`，设置：
   ```
   EMBEDDER=clip
   INSTALL_ML=true
   ```
2. 重新构建并启动 `worker` 服务（`INSTALL_ML=true` 会让 `server/Dockerfile` 额外执行 `pip install -e ".[dev,ml]"`，安装 `torch` + `cn-clip` + `pillow`，体积达数 GB，构建耗时明显增加）：
   ```bash
   docker compose up -d --build worker
   ```
3. `worker` 启动后，`app.workers.matcher.run_match`（ARQ 任务入口）会按 `settings.embedder`（读取 `EMBEDDER` 环境变量）选择 `ChineseClipEmbedder`（`cn_clip` ViT-B-16，CPU 推理，图片下载失败自动重试）而非 `MockEmbedder`。
4. 注意：`api` 服务的 build 不受影响，始终 `INSTALL_ML=false`（`api` 不做匹配计算，无需装 ML 依赖）；`/match/start?sync=true` 的同步路径出于演示/测试目的固定用 `MockEmbedder`，真实 CLIP 只在异步入队（`sync=false`，由 `worker` 消费）路径生效。

## 常见问题

- **`/match/start` 返回 `queued` 后候选一直不出现？** 确认 `worker` 容器已启动且能连上 Redis/Postgres（`docker compose logs worker`），用 `GET /match/monitor` 看 `match_status` 是否推进；mock 链路下应该秒级/分钟级跑完，取决于商品数量。
- **候选数量比预期少？** 检查 `match_stats.platforms_skipped`（账号池当天无可用账号/限速命中）与 `platforms_failed`（平台请求异常，如反爬拦截触发 `report_risk` 冷却）。
- **设置了 `EMBEDDER=clip` 但候选看起来还是 mock 向量在起作用？** 确认改的是 `worker` 服务的环境（`.env` 里的 `EMBEDDER`），并且是异步入队路径（`sync=false`）——`sync=true` 的同步演示路径固定使用 mock。
- **重新 `docker compose up --build` 后 worker 镜像还是很快、没装 torch？** 确认 `.env` 里 `INSTALL_ML=true` 且执行了 `docker compose up -d --build worker`（不带 `--build` 不会重新触发 `ARG INSTALL_ML` 的条件安装）。
