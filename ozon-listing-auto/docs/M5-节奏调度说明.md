# M5 使用说明：上架节奏调度（配节奏 → 排期 → tick 逐一挂靠 → 监控）

本说明面向已完成 M1 采集、M2 货源匹配、M3 评分审核、M4 定价与草稿确认的用户，走一遍 M5 上架节奏调度的完整操作流程，并说明如何切到生产用的跨进程实时进度与真实 Ozon 挂靠。

## 0. 前置说明：M5 是「排期 + 限速」的另一条上架路径，M4 直发路径仍保留

- M4 的 `POST /listing/publish` 是「确认后立即把该任务下所有 `confirmed` 草稿逐条挂靠完」，没有排队和限速，一次请求内（或一次 worker 任务内）全部挂完。
- M5 在 `confirmed` 之后加了一层**节奏调度**：先给每条草稿排出 `scheduled_at`（随机间隔 + 活跃时段 + 每日上限），再按周期 tick 一次只挂靠到期的一条（或按 `wait_ozon_approval` 等上一条 Ozon 审核通过后才继续）——用于避免短时间内批量创建大量跟卖 offer 触发 Ozon 风控。
- 两条路径**互不冲突、并存**：同一批 `confirmed` 草稿，你可以选择直接调 `/listing/publish` 全量挂靠，也可以先 `/publish/schedule` 排期再靠 cron/`tick` 慢慢推进；一旦某条草稿被 `/publish/schedule` 排入 `scheduled` 状态，就不再是 `confirmed`，`/listing/publish` 不会再处理它（它只处理 `confirmed` 状态的草稿）。
- 与 M2/M3/M4 一致，M5 的验收标准也是「`docker compose up`（mock Ozon 写入 + memory 进度后端）一键跑通」，无需任何真实 Ozon 凭据或额外 Redis 配置即可走完「配节奏 → 排期 → tick → 监控」全链路。

## 1. 完整流程

```
采集(M1) → 匹配(M2) → 评分(M3) → 审核采用(M3)
  → 建店铺(M4) → 配定价(M4) → 生成草稿(M4: POST /listing/build)
    → 草稿确认(M4: POST /listing/{id}/confirm 或 auto-confirm, draft → confirmed)
      → 配节奏(M5: PUT /pace 或 PublishMonitor 页面「节奏配置」表单)
        → 排期(M5: POST /publish/schedule, confirmed → scheduled, 排出 scheduled_at)
          → 周期 tick(M5: ARQ cron 每分钟自动跑, 或手动 POST /publish/tick)
            逐一挂靠(scheduled → pending_review → published/failed, 等审核门见第 3 节)
              → PublishMonitor 页面(/monitor)看队列分布/下一条 ETA/实时更新
                → (可选)生产环境切 PROGRESS_BACKEND=redis + OZON_SELLER_PROVIDER=real, 见第 5 节
```

## 2. 配节奏

节奏配置存在 `publish_pace` 表，支持按任务覆盖或写一条 `task_id=null` 的全局默认。调用方（operator 及以上）：

```
PUT /pace?task_id=<id>
{"min_interval_sec": 60, "max_interval_sec": 180, "daily_limit": 200,
 "active_hours": [9, 23], "wait_ozon_approval": true}

GET /pace?task_id=<id>   # 未单独配置该任务时按「任务级 → 全局默认(task_id=null) → 内置默认」三级回退
```

字段含义：

| 字段 | 说明 | 默认值 |
| --- | --- | --- |
| `min_interval_sec` / `max_interval_sec` | 两条草稿排期时间之间的随机等待区间（秒），排期时按 `random.randint(min, max)` 取值累加 | 60 / 180 |
| `daily_limit` | 每日排入的上架条数上限；同一天已排满则顺延到次日活跃时段起点；`≤0` 视为不限（避免误配 0 导致排期死循环） | 200 |
| `active_hours` | 活跃时段 `[start_hour, end_hour]`（24 小时制），排期时间落在时段外会被挪到下一个时段起点（`next_active_window()`） | `[9, 23]` |
| `wait_ozon_approval` | tick 时是否要等上一批已提交 Ozon 审核的草稿（`pending_review`）出结果（`approved`/`rejected`）后才推进下一条 | `true` |

前端「上架监控」页（`/monitor`，`PublishMonitor.tsx`）的「节奏配置」表单即读写这个接口，输入任务 ID 后自动加载当前生效的节奏、点「保存节奏」写入。

## 3. 排期：把 `confirmed` 草稿排出 `scheduled_at`

```
POST /publish/schedule?task_id=<id>   →   {"scheduled": <排入条数>}
```

`plan_schedule()`（`app/services/publish_scheduler.py`）对该任务下所有 `status=confirmed` 且 `scheduled_at is null` 的草稿，按 `id` 升序逐条：

1. 在上一条排定时间基础上累加一个 `[min_interval_sec, max_interval_sec]` 区间内的随机秒数；
2. 若落在 `active_hours` 时段外，挪到下一个时段起点（跨天则挪到次日）；
3. 若当天（按挪好后的日期）已排入条数达到 `daily_limit`，顺延到次日活跃时段起点，重新计数；
4. 写入 `scheduled_at`，草稿状态 `confirmed → scheduled`。

`plan_schedule` 可反复调用：已经排过期（`scheduled_at` 非空）的草稿不会被重新排期，只处理新产生的 `confirmed` 草稿，因此「build → confirm → schedule」可以按批次多次执行。

## 4. 周期 tick：逐一挂靠 + 等审核门

### 4.1 tick 做什么

`tick_publish()`（`app/workers/publisher.py`）每次调用：

1. **等审核门**（`wait_ozon_approval=true` 时）：先查该任务下所有 `pending_review`（已提交等 Ozon 审核）的草稿，逐条调用 `OzonSellerProvider.get_product_status()`：
   - 返回 `approved` → 该草稿转 `published`；
   - 返回 `rejected` → 该草稿转 `failed`，`error="ozon rejected"`；
   - 其余（如 `pending`）→ 标记本轮「仍在等待」。
   只要有一条仍在等待，本轮 tick 到此为止，**不会**再去挂靠下一条待排期草稿——避免上一条还没审核通过就抢跑，短时间内堆积过多同批 offer。
2. **挂靠下一条**：等审核门通过（或 `wait_ozon_approval=false`）后，取该任务下 `scheduled_at <= now` 的到期草稿（按 `scheduled_at` 升序，默认 `max_batch=1` 即每次只挂一条）逐条调用 `create_follow_offer()`：
   - 成功：`wait_ozon_approval=true` 时置为 `pending_review`（等下一轮 tick 轮询审核结果），否则直接置为 `published`；
   - 失败：置为 `failed` 并记录 `error`。
   每条草稿独立数据库 session + 独立 `try/except`，单条异常不影响同批其余草稿（与 M4 `run_publish_core` 一致的隔离方式）。

### 4.2 两种触发方式

- **周期自动**（默认，推荐）：`app/workers/arq_worker.py` 里注册了 ARQ cron 任务 `run_publish_tick`，`cron_jobs = [cron(run_publish_tick, minute=set(range(0, 60)))]` —— **每分钟触发一次**，扫描所有存在到期 `scheduled` 草稿或 `pending_review` 草稿的任务，逐个 `tick_publish`。这是生产环境下推荐的用法：配好节奏 + 排完期后不用管，cron 会持续推进。
- **手动触发**：`POST /publish/tick?task_id=<id>&sync=true|false`（publisher/admin 权限）：
  - `sync=true`：请求内**同步**跑一次 `tick_publish`（**固定用 `MockOzonSeller`**，便于本地/演示/测试立即看到一条草稿被处理的效果），返回 `{"published": n, "pending_review": n, "failed": n, "waiting": bool}`；
  - `sync=false`（默认）：不直接执行，而是经 ARQ 立即入队一次 `run_publish_tick`（按 `OZON_SELLER_PROVIDER` 选真实/mock seller），返回 `{"status": "queued"}`，效果需另外看 `/publish/monitor` 或等下一轮 cron。

前端「上架监控」页的「手动上架一条」按钮调用的是 `sync=true` 路径（`web/src/api/publish.ts::tick`），仅用于演示/联调时立即触发一次，**不会**对接真实 Ozon（见第 5 节）。

## 5. 监控

```
GET /publish/monitor?task_id=<id>
→ {"counts": {"draft": .., "confirmed": .., "scheduled": .., "pending_review": .., "published": .., "failed": ..},
   "next_scheduled_at": "2026-07-18T10:00:00+00:00" | null,
   "pace": {...当前生效的节奏配置...}}
```

前端「上架监控」页（`/monitor`）展示：

- 队列统计卡片：草稿 / 已确认 / 已排期 / 审核中 / 已上架 / 失败，按状态分组的草稿数量；
- 「队列监控 · 下一条」标题栏展示 `next_scheduled_at`（下一条待上架草稿的排定时间）；
- 「开始排期」「手动上架一条」「刷新」按钮，以及节奏配置表单。

### 5.1 实时更新（首个使用 WebSocket 的前端页面）

`tick_publish` 每**挂靠**一条草稿（第 4.1 节的第 2 步）会通过 `Broadcaster.publish()` 广播一条进度消息（`{"task_id", "draft_id", "phase": "publish", "published", "pending_review", "failed"}`）；第 1 步的等审核门轮询本身不广播。注意 M4 的直发路径 `run_publish_core`（`/listing/publish`）目前不接 `Broadcaster`，不会推送 WS 进度——WS 进度推送目前只有 `tick_publish` 这一条路径会触发。PublishMonitor 页面连接 `ws://<host>/ws/progress`（复用 M1 起就存在的进度 WS 端点，M5 是第一个真正消费它的前端页面）：收到任意消息就重新拉取一次 `/publish/monitor`，实现近实时刷新。**WS 连接失败、出错或断开**（`onerror`/`onclose` 均处理，避免只处理 `onerror` 导致正常断开后静默停更）**都会回退为每 5 秒轮询** `GET /publish/monitor`，保证页面在 WS 不可用时依然能看到进度更新，只是延迟变高。

## 6. 生产环境：跨进程实时进度 + 真实 Ozon 挂靠

本地/CI 默认用 `PROGRESS_BACKEND=memory`（单进程本地 fan-out）+ `OZON_SELLER_PROVIDER=mock`（确定性挂靠成功），`docker compose up` 一键即可跑通全链路演示，不需要下面这些配置。生产环境建议按需开启：

### 6.1 `PROGRESS_BACKEND=redis`（跨进程实时进度）

- **问题**：ARQ cron 触发的 `tick_publish` 跑在 `worker` 容器/进程里，而 WebSocket 连接是被 `api` 容器/进程持有的——`memory` 后端下 `Broadcaster.publish()` 只在当前进程内 fan-out，`worker` 进程广播的消息永远推不到 `api` 进程持有的 WS 连接上，PublishMonitor 页面就只能靠 5 秒轮询兜底，感知不到「实时」。
- **开法**：设置环境变量 `PROGRESS_BACKEND=redis`（`app/core/config.py::Settings.progress_backend`）。之后：
  - `Broadcaster.publish()` 改为把消息 `PUBLISH` 到 Redis 频道 `ws:progress`（`app/core/progress.py::CHANNEL`）；
  - `api` 进程在 FastAPI `lifespan` 启动时额外起一个后台协程 `broadcaster.start_redis_subscriber()`，订阅该频道，收到消息后在本进程内 `_local_broadcast()` 给自己持有的所有 WS 连接。
- 这样无论 tick 是被哪个进程（`worker` cron 还是 API 进程内 `sync=true` 手动触发）执行的，进度都会经 Redis 中转推到所有 `api` 副本持有的 WS 连接上——多副本 `api` 水平扩展场景下也适用。
- 代价：多一次 Redis 网络往返；`memory` 后端在单进程/单副本部署（或本地开发）下已经够用，没有跨进程需求时不必开启。

### 6.2 `OZON_SELLER_PROVIDER=real`（真实挂靠 + 审核状态轮询）

- 设置环境变量 `OZON_SELLER_PROVIDER=real`（默认 `mock`）。**只影响异步/worker 路径**：`POST /publish/tick?sync=false` 入队后由 `run_publish_tick`（cron 或手动触发一次）执行，以及 M4 的 `POST /listing/publish?sync=false` → `run_publish`，两者都会用 `get_ozon_seller(settings.ozon_seller_provider)` 选 seller。**`sync=true` 的同步路径（`/publish/tick?sync=true` 与 `/listing/publish?sync=true`）当前仍固定使用 `mock`**，是刻意保留的「本地/演示/测试」捷径，即便生产环境把 `OZON_SELLER_PROVIDER` 设为 `real`，用 `sync=true` 手动点一下也不会误触真实 Ozon 写入；要走真实挂靠必须用 `sync=false`（默认）配合 cron/worker。
- 生产用真实挂靠前需先在「店铺管理」（`/shops`）配好真实 `client_id`/`api_key`，并把生成的草稿关联到该店铺（`shop_id`）。
- **`RealOzonSeller.get_product_status()`（M5 新增）当前是占位实现**：请求 `https://api-seller.ozon.ru/v2/product/info`（端点未经 live 联调校正），且无论响应内容如何都恒返回 `"pending"`（`app/services/ozon_seller/real.py`）——也就是说，即便设置了 `OZON_SELLER_PROVIDER=real` 且 `wait_ozon_approval=true`，"等审核门"目前**永远等不到 `approved`/`rejected`**，草稿会一直卡在 `pending_review`，tick 不会再推进下一条。要真正用上等审核门 + 真实挂靠，必须在 live 联调时把 `get_product_status()` 改成解析 Ozon 真实返回、正确映射到 `approved`/`pending`/`rejected` 三态；在此之前，真实环境如需继续推进队列，建议把 `wait_ozon_approval` 设为 `false`（不等审核直接继续，牺牲「上一条通过再上下一条」的保护）。
- `RealOzonSeller.create_follow_offer()` 同样是占位实现（`_ENDPOINT` 指向 `product/import`），真实跟卖端点字段需在联调时校正，详见 [`docs/M4-定价上架说明.md`](M4-定价上架说明.md) 第 5 节。

## 常见问题

- **`/publish/schedule` 返回 `scheduled: 0`？** 确认任务下有草稿处于 `confirmed` 状态（M4 步骤：先 `/listing/build` 生成草稿，再 `/listing/{id}/confirm` 或 `/listing/auto-confirm` 确认）；已经排过期的草稿（`scheduled_at` 非空）不会被重复排期。
- **`/publish/tick` 一直返回 `published: 0` 且 `waiting: true`？** 说明有草稿卡在 `pending_review` 且 `wait_ozon_approval=true`，等审核门没通过——本地/CI 用 `mock` seller 默认 `get_product_status` 返回 `approved` 会自动放行；若生产环境切了 `OZON_SELLER_PROVIDER=real`，见第 6.2 节的已知限制（占位实现恒返回 `pending`）。
- **`/publish/tick` 返回 `published: 0` 但也不是 `waiting`？** 检查是否有 `scheduled_at <= now` 的到期草稿——排期时间在未来的草稿不会被本轮 tick 处理，等到期或调整节奏/等 cron 下一轮。
- **cron 好像没有自动 tick？** 确认 `worker` 容器正常运行且能连上 Redis（ARQ cron 依赖 `worker` 进程内的调度循环，不是 API 进程），以及对应任务确实存在到期的 `scheduled`/`pending_review` 草稿（`run_publish_tick` 只扫有这两种状态草稿的任务）。
- **PublishMonitor 页面数据不实时更新？** 默认 `memory` 进度后端下，只有触发 tick 的那个进程能广播给它自己持有的 WS 连接；如果 tick 是由另一个 `worker` 进程的 cron 触发的，`memory` 后端广播不到 `api` 进程的 WS，页面会退化成 5 秒轮询（仍会更新，只是不「实时」）。要做到跨进程实时，见第 6.1 节切 `PROGRESS_BACKEND=redis`。
- **想验证真实挂靠效果，配了真实店铺凭据后 `/publish/tick` 还是走 mock？** 检查是否用的是 `sync=true`——如第 6.2 节所述，`sync=true` 无论 `OZON_SELLER_PROVIDER` 设成什么都固定走 `mock`，必须用 `sync=false`（配合 cron 或手动入队一次）才会按 `OZON_SELLER_PROVIDER` 选真实 seller。
