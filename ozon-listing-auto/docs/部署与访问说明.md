# 部署与访问说明（M7：本地测试栈 → 生产部署）

本说明面向已完成 M1~M6 全链路开发、准备验收或上线的用户，说明两套 compose 栈的区别、如何在本地跑一遍完整业务流程做验收，以及如何把系统部署到带域名的公网服务器（HTTPS）。

## 0. 两套栈的区别

| | `docker-compose.yml` | `docker-compose.prod.yml` |
| --- | --- | --- |
| 定位 | 本地测试/开发栈 | 生产栈 |
| 协议 | HTTP | HTTPS（Nginx 终结 TLS） |
| 域名/证书 | 不需要 | 需要域名 + Let's Encrypt 证书（`deploy/certbot-init.sh`） |
| 端口暴露 | db/redis/api/web 均发布到宿主机端口 | 仅 `nginx` 发布 80/443，db/redis/api/worker **不发布公网端口** |
| 环境变量文件 | `.env` | `.env.prod` |

**建议流程**：先用本地测试栈把整条业务链路（建任务 → 采集 → 匹配 → 评分 → 审核 → 定价 → 上架 → 监控）跑通验收，确认无误后再按第 2 节起生产栈上线。两套栈相互独立，互不影响。

## 1. 本地测试（验收基线）

```bash
cd ozon-listing-auto
cp .env.example .env   # 按需修改，默认值即可跑通全流程
docker compose up -d --build
```

浏览器打开 `http://localhost:8080`，用默认账号登录：

```
用户名：admin
密码：admin123
```

（可通过 `.env` 里的 `ADMIN_USER`/`ADMIN_PASSWORD` 覆盖）

按下列顺序过一遍全流程，确认每一步都符合预期：

1. **建任务**（跟卖 `follow` 或自建 `create`，见任务中心）→ **启动采集**（默认 `provider=mock`，无需外部依赖）。
2. **货源匹配**（`/match/start`）→ 在候选列表里看到多平台候选、去重代表。
3. **评分**（`/score/start`）→ 在**审核台**（ReviewBoard）逐个采用/拒绝候选，或按阈值自动采用。
4. **定价**（`/settings/pricing` 配好参数）→ **生成草稿**（`POST /listing/build`）。
   - 自建（`create`）分支额外需要：在**图片工作室**（`/image-studio`）触发改图并采用产物；在**上架审核**（`/listing`）确认类目/属性（`POST /listing/{id}/confirm-category`）。
5. **确认草稿**（`/listing/{id}/confirm` 或按阈值自动确认）→ **上架**：可直接 `/listing/publish` 全量挂靠，或配好节奏（`/pace`）后走 `/publish/schedule` + `/publish/tick` 排期挂靠。
6. **上架监控**（`/monitor`）→ 查看队列分布（草稿/已确认/已排期/审核中/已上架/失败）与实时 WS 更新。

全部走通即代表本地测试栈验收通过，可以进入下一节的生产部署。

停止本地测试栈：`docker compose down`（加 `-v` 一并清空数据库数据卷）。

## 2. 服务器准备

- 安装 Docker + Docker Compose（v2，`docker compose` 子命令）。
- 准备一个域名，把它的 **A 记录解析到服务器公网 IP**（生产部署需要真实域名签发 Let's Encrypt 证书，不支持裸 IP）。
- 云安全组 / 防火墙放行 **80、443** 端口（HTTP-01 证书挑战走 80，业务流量走 443；SSH 端口按需另外放行）。

## 3. 首次签证

复制生产环境变量样例并按实际填写：

```bash
cd ozon-listing-auto
cp .env.prod.example .env.prod
```

编辑 `.env.prod`，至少要改：

| 变量 | 说明 |
| --- | --- |
| `DOMAIN` | 已解析到本机公网 IP 的域名 |
| `CERTBOT_EMAIL` | Let's Encrypt 证书到期提醒邮箱 |
| `POSTGRES_PASSWORD` | 数据库密码（改强密码，`DATABASE_URL` 里的密码要同步改） |
| `JWT_SECRET` | **务必重新生成**（不要沿用样例里的占位值） |
| `FERNET_KEY` | **务必重新生成**：`python -c "from cryptography.fernet import Fernet;print(Fernet.generate_key().decode())"` |
| `CORS_ORIGINS` | JSON 数组字符串，收紧为实际前端域名，如 `["https://your-domain.com"]` |
| `ADMIN_PASSWORD` | 首个管理员账号密码，改强密码 |
| `OZON_SELLER_PROVIDER`/`LLM_PROVIDER`/`EMBEDDER`/`IMAGE_PROVIDER` | provider 开关，默认全 `mock`，按需切真实并配相应凭据（见各里程碑说明文档） |

填好后执行首次签证脚本（**此时全栈尚未启动**，`certbot --standalone` 会临时占用本机 80 端口完成 HTTP-01 挑战，从而避开「nginx 需要证书才能启动、证书又需要 nginx 提供服务来完成挑战」的先有鸡还是先有蛋问题；脚本内部的 `docker compose` 调用已带 `--env-file .env.prod`）：

```bash
DOMAIN=<你的域名> CERTBOT_EMAIL=<你的邮箱> sh deploy/certbot-init.sh
```

脚本会先尝试停掉可能已在跑的 `nginx` 容器（占用 80 端口），再用 `docker compose ... run --rm -p 80:80 --entrypoint certbot certbot certonly --standalone -d "$DOMAIN" --email "$CERTBOT_EMAIL" --agree-tos --no-eff-email` 签发证书，证书落到 `letsencrypt` 卷（`/etc/letsencrypt/live/$DOMAIN/`）供后续 nginx 容器加载。

## 4. 起全栈

```bash
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d --build
```

首次启动会和本地测试栈一样自动执行 `alembic upgrade head` 建库、幂等创建首个管理员账号。浏览器打开 `https://<你的域名>`，用 `.env.prod` 里配置的管理员账号登录，按第 1 节的流程再验证一遍。

`docker-compose.prod.yml` 中各服务：

- `db`（Postgres+pgvector）、`redis`：仅内网可达，不发布端口。
- `api`、`worker`：读 `.env.prod`（`env_file`），不发布端口，改图产物写到共享卷 `static`。
- `nginx`（`web` 镜像 build，挂载 `deploy/nginx.prod.conf` 为模板）：发布 80/443，80 端口跳转 HTTPS 并承载 ACME 续期挑战，443 端口 TLS 终结 + 反代 `/api`、`/ws` 到内部 `api:8000`、其余走前端静态文件 `try_files`。
- `certbot`：后台循环 `certbot renew --webroot -w /var/www/certbot`，每 12 小时尝试一次续期，证书与 webroot 通过卷与 nginx 共享。

## 5. 安全加固清单

- **强制 HTTPS + HSTS**：`deploy/nginx.prod.conf` 中 80 端口除 ACME 挑战外全部 301 跳转到 `https://`；443 响应头带 `Strict-Transport-Security: max-age=63072000`，以及 `X-Content-Type-Options`/`X-Frame-Options`/`Referrer-Policy`。
- **内部端口不公开**：`docker-compose.prod.yml` 中 db/redis/api/worker 均未声明 `ports:`，只能通过 compose 内网访问；公网仅暴露 nginx 的 80/443。
- **登录失败限流**：按 `用户名|IP` 维度滑动窗口计数，默认 `LOGIN_MAX_ATTEMPTS=5` 次失败在 `LOGIN_WINDOW_SEC=300` 秒窗口内触发，锁定 `LOGIN_LOCKOUT_SEC=900` 秒（15 分钟），`/auth/login` 锁定期间返回 `429` + `Retry-After`。
- **CORS 收紧**：`.env.prod` 的 `CORS_ORIGINS` 改为实际前端域名（部署态前端与后端同源经 nginx 反代，此为纵深防御；`allow_credentials` 恒 `false`）。
- **密钥已重新生成**：`JWT_SECRET`、`FERNET_KEY`、`POSTGRES_PASSWORD`、`ADMIN_PASSWORD` 均不沿用开发/样例默认值。
- （可选）云安全组仅放行 80/443/SSH，其余端口一律不对公网开放。

## 6. 更新方式

```bash
git pull
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d --build
```

`up -d --build` 会按需重建变更的镜像并滚动替换容器，其余服务不受影响；数据库/静态资源等持久化数据在卷里，不会因为重建容器丢失。更新完成后员工只需**刷新浏览器**（前端为静态资源，nginx 会提供新版本），无需额外操作。

## 7. 证书续期与排查

- `certbot` 容器已在后台循环执行 `certbot renew --webroot -w /var/www/certbot`（每 12 小时检查一次），Let's Encrypt 证书 90 天有效期内会在到期前自动续期，无需人工触发。
- 排查续期是否正常：

  ```bash
  docker compose -f docker-compose.prod.yml logs certbot
  ```

- 若证书文件更新但 nginx 未加载新证书，可重载 nginx 配置（无需重启容器、不中断连接）：

  ```bash
  docker compose -f docker-compose.prod.yml exec nginx nginx -s reload
  ```

- 若首次签证失败（如域名未解析到位、80 端口被占用），排查后重新执行第 3 节的 `deploy/certbot-init.sh` 即可，脚本可重复执行。

## 8. 已知 live 后置项

以下能力在当前 mock-first 阶段仅有占位实现，真实上线到 Ozon 生产环境前需要联调校正（详见各自里程碑说明文档）：

- **`RealOzonSeller`**（`app/services/ozon_seller/real.py`）：`create_follow_offer`/`create_product`/`get_product_status` 的接口地址、请求体字段、审核状态映射均为占位，需按 Ozon Seller API 真实文档联调校正（详见 [`docs/M4-定价上架说明.md`](M4-定价上架说明.md)、[`docs/M5-节奏调度说明.md`](M5-节奏调度说明.md)）。
- **`RealCategoryTree`**（`app/services/category_tree.py`）：真实类目树（走 composer-api `categoryChildV3`）未实现，仅占位 import，当前只有 `mock` 固定小树可用。
- **外部生图 provider**（`openai_compat`/`http`，`app/services/imagegen/`）：方法体为 `NotImplementedError` 占位，需接入真实 AI 生图服务后才可用；`rmbg` 去背景需额外安装 `rembg`（`INSTALL_ML=true`）才有真实效果，否则自动降级为白底。
- **真实爬虫 cookie / 代理**：`provider=composer`（M1 采集）与 1688 图搜/关键词搜索（M2 货源匹配）依赖配置中心里可用的 cookie / 代理才能稳定拿到数据，反爬策略随时间变化，需持续维护；拼多多真实抓取（`selenium` + 代理截获）尚未接入。
- **改图产物尚不能公网访问**：改图产物落盘到 `api`/`worker` 共享卷 `static`，由 `api` 进程内部通过 `/static` 提供（容器内可访问），但当前 nginx 反代（`deploy/nginx.prod.conf`）**没有** `location /static/` 反代规则，`/static/...` 请求会落到 `location /` 的 `try_files … /index.html`，即公网并不能直达 `https://<域名>/static/...`。若要接 Ozon 拉图上架（需要公网可达的图片 URL），需先在 nginx 增加对应 `location /static/` 反代到 `api:8000`，属 live 后置项，接 Ozon 拉图时再补。
- **多 `api` 实例的登录限流**：`LoginThrottle` 当前是进程内内存计数，单机单 `api` 实例下限流生效；若后续水平扩展多个 `api` 实例，需要把限流计数迁到 Redis 等共享后端才能保证跨实例一致。
- **多 `api` 实例的 WS 进度广播**：`PROGRESS_BACKEND=memory` 时进度广播是进程内 fan-out，本生产部署包默认单 `api`/`worker` 实例；若后续需要多实例横向扩展，需设置 `PROGRESS_BACKEND=redis`（详见 [`docs/M5-节奏调度说明.md`](M5-节奏调度说明.md)）。
