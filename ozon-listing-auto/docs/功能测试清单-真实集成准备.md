## 一、AI 模型接入（LLM）✅

**用途**：商品标题/描述文案生成、类目建议、关键词等。

**准备**：一个 OpenAI 兼容的大模型服务（默认预设是阿里云百炼 qwen）：
- `api_key`
- `base_url`（如百炼 `https://dashscope.aliyuncs.com/compatible-mode/v1`，或你自己的兼容端点）
- 模型名（如 `qwen-plus` / `gpt-4o-mini` 等）

**操作**：配置页「LLM 配置」→ Provider 选 `openai` → 填 base_url / api_key / 模型 → 保存。
**验证**：进「审核台/图片工作室」等触发文案生成的地方，看是否返回真实模型内容（非 mock 固定串）。

---

## 二、生图（文生图 / 改图）✅

**用途**：AI 生成主图 / 按 prompt 改图。

**准备**（二选一）：
- **openai_compat 类**：`img_api_key` + `img_base_url` + `img_model`（图像生成模型）。
- **http 通用类**：`img_base_url` + `img_request_template`（请求体 JSON 模板）+ `img_response_path`（响应里取图 URL/base64 的点路径）。
- 本地改图（裁剪/加水印等）恒走本地 Pillow，无需外部服务。

**操作**：配置页「AI 生图配置」→ Provider 选 `openai_compat` 或 `http` → 填对应字段 → 保存。可配 `fallback` 降级顺序。
**验证**：进「图片工作室」触发生图/改图，看产出的是真实新图而非占位图。

---

## 三、类目树浏览（真实 Ozon 类目）✅

**用途**：浏览/选择 Ozon 真实类目、类目属性映射。

**准备**：
- **Ozon 站点登录 Cookie**（在浏览器登录 ozon.ru 后从开发者工具复制整条 Cookie）。
- 可选：代理（Ozon 对境外/机房 IP 有风控，建议住宅/干净代理）。

**操作**：
1. 系统设置 → `category_tree_provider` 改为 `real`。
2. 爬虫配置 → 填 `cookie`（+ 可选 `proxy`、超时、请求间隔、重试）。
**验证**：类目浏览页能拉出真实 Ozon 类目层级；被风控时会报 blocked（换 Cookie/代理/降速）。

> ⚠️ Cookie 会过期；风控（429/403）时系统会退避重试。CAPTCHA 需人工在浏览器过。

## 五、1688 采集 ⚠️（结构就绪，需你补真实接口）

**现状**：`Ali1688Provider` 是**配置驱动**的骨架——端点、请求方法、额外参数/头、响应路径都能配，Cookie 账号池也就绪；但 **1688 mtop 的真实签名（sign）没有逆向**。所以要真跑，需要你提供**已能直接请求成功的搜索接口**（URL + 参数），系统按配置去调。

**准备**：
1. **1688 登录 Cookie**（登录后从浏览器抓）。
2. **真实搜索接口信息**（你从 1688 网页/APP 抓包得到）：
   - 图搜端点 URL、关键词搜索端点 URL
   - 请求方法（GET/POST）
   - 额外请求参数（JSON，含 sign/token 等）、额外请求头（JSON）
   - 响应里 offerList 的点路径（如 `data.offerList`）

**操作**：
1. 配置页「货源配置」→ 填 `ali1688_*` 各端点/方法/额外参数·头/offerList 路径。
2. **添加 1688 Cookie 账号**：前端「货源账号」页（`/source-accounts`，admin）可视化新增，平台选「1688」、粘贴 Cookie 即可；同页支持启停/更新 Cookie/冷却手动恢复/删除。
**验证**：在采集/匹配流程选 1688 源，看是否返回真实 offer；被风控时账号自动进冷却换号。

> 若你能抓到"带签名、能直接 curl 通"的接口，我就能把它配进来真跑；否则 1688 真实采集需要额外做签名逆向或走浏览器自动化（工作量较大，当前后置）。

## 六、CLIP 图片去重匹配 ✅（需重建镜像）

## 七、自动上品到 Ozon（Seller API）✅ 跟卖+自建均可真发

**现状**：`RealOzonSeller` 已按官方 `api-seller.ozon.ru` 实现（跟卖 `import-by-sku`、自建 `v3/product/import`、状态 `import/info`，Client-Id+Api-Key 头），且发布接口 `POST /listing/publish` / `POST /publish/tick` **已接线**——不论 `sync=true`（请求内同步跑）还是 `sync=false`（异步/worker/cron），都经 `resolve_seller()` 按系统设置的 `ozon_seller_provider`（`mock`/`real`）选 provider，不再固定 mock。

**7.1 跟卖上品（Seller API）✅**

**操作**：
1. 「店铺管理」（`/shops`）新增/编辑店铺，填真实 `Client-Id` + `Api-Key`（**建议先用 Ozon 沙箱环境测**，避免真发错货）。
2. 系统设置（`/settings/system`，admin）「上品模式」切到**真实（调用 Ozon）**。
3. **「试运行(dry-run)」开关**：默认开启——切到真实后先保持开启，此时会构造真实 Ozon 请求但**不提交**，用于安全演练/核对请求体；确认无误后关闭试运行，才会真正提交发出。
4. 正常走上架审核台确认草稿 → 挂靠上架（`/listing/publish` 或按节奏 `/publish/tick`）。

**验证**：草稿状态变为 `published` 并回写真实 `ozon_result.ozon_product_id`（试运行状态下会返回构造的请求但不落 Ozon 侧真实商品）；跟卖走通需要一个**真实存在、允许跟卖的目标 Ozon SKU**。

**7.2 自建上品（v3/product/import）✅（需先补充信息）**

**现状**：自建草稿（`listing_drafts`）已扩了 `type_id`（该类目下商品类型）+ 尺寸(`depth`/`width`/`height`)/`weight`/单位字段，并新增 Ozon 卖家端类目查询 `GET /ozon-catalog/types|attributes|attribute-values`（`system.ozon_seller_provider=real` 且已配真实店铺时走真实类目/类型/属性接口，否则样例数据）供前端下拉选择。

**操作**：
1. 同上先完成 7.1 的 1~3 步（店铺凭据 + 真实模式 + 试运行开关）。
2. 在「上架审核」页对该自建草稿的「补充信息」表单里，选好**类型（type_id）**、填**尺寸/重量**、选齐**必填属性**（属性值来自 `/ozon-catalog/attribute-values` 的字典值，保存后走 `POST /listing/{draft_id}/confirm-create-fields` 回写）。
3. 未补齐上述信息时，`POST /listing/{id}/confirm`（确认上架）会被拒绝，提示「自建缺类型或尺寸/重量, 请先补充」——这是硬性闸门，不是可选步骤。
4. 补齐后正常确认 → 挂靠/建品。

**已知限制**：草稿 `attributes` 字典的键必须是数字型 Ozon attribute id；填了非数字键会在挂靠时使该草稿被标记为 `failed`（记录具体错误，不会静默丢弃），需要核对属性 id 后重新提交。

**准备**：
- 每个店铺的 **Client-Id + Api-Key**（Ozon 卖家后台 → API 设置）。
- 入口：「店铺管理」页新增店铺时填 `client_id` / `api_key`（+ 沙箱开关）。
