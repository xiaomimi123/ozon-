# 真实爬虫接入说明（composer-api 真实抓取）

本说明面向已完成 M1 采集骨架的用户，聚焦「真实爬虫接入」这个非里程碑（cross-cutting）任务补的能力：`provider=composer` 从占位实现变为可配置 cookie/代理的真实 Ozon composer-api 抓取，含反爬硬化、真实响应解析、Seller 挂靠 real/mock 切换、`@live` 冒烟测试。

## 1. 目的

M1 起 `provider=composer` 就存在，但此前 cookie/代理无处配置、解析层未对齐真实响应结构，实际不可用。本次改动补齐三块：

1. **配置**：后台 `/settings/crawler`（admin）可视化填 cookie/代理/超时/间隔/重试，替代此前只能改代码/环境变量的方式。
2. **硬化**：请求层注入 cookie 头、随机 UA、间隔抖动，遇 307/301/302/403/429 反爬信号指数退避重试，耗尽后抛出**可操作**的错误（提示去更新 cookie/代理），采集任务捕获后置为 `failed` 并把错误写进 `stats.error`，不会静默吞掉。
3. **解析**：`parser.py` 对齐了真实 Ozon 前台返回的 `widgetStates` 结构（此前是占位/猜测结构），用真实抓包样本做夹具驱动单测。

## 2. Cookie 获取步骤（关键前置：Ozon 有滑块反机器人验证码）

**实测结论**：Ozon 前台对干净会话（新 IP/无历史 cookie）几乎必定弹出滑块验证码拦截。composer-api 是前台页面内部调用的接口，同样受这层防护影响。因此**真实采集不是配好 cookie 就一定能跑，必须先有一个已经在浏览器里人工通过一次验证码的会话**，再把该会话的 cookie 复制出来使用。

获取步骤：

1. 用真实浏览器打开 `https://www.ozon.ru`，如遇滑块验证码，人工完成一次。
2. 打开浏览器 DevTools → Network 面板，刷新页面或做一次搜索，任选一个 `ozon.ru` 域下的请求。
3. 在请求头里找到 `Cookie` 字段，复制**完整值**（形如 `abc=1; def=2; __Secure-ETC=...`）。
4. 登录本系统后台管理员账号，进入 `/settings/crawler`，把复制的 Cookie 粘贴进「cookie」输入框保存。

Cookie 有时效性，验证码状态/风控信号会随时间失效——采集任务若开始持续失败（见第 5 节），重复上述步骤换一次新 cookie。

## 3. 代理配置

Ozon 对非俄罗斯（RU）出口 IP 的访问更容易触发风控/验证码，**建议配 RU 出口的代理**（`/settings/crawler` 的「proxy」字段，格式为 `http://user:pass@host:port` 或 `socks5://host:port`，即 `httpx.AsyncClient(proxy=...)` 支持的格式）。代理与 cookie 是两个独立的风控维度，二者都配置能显著提高稳定性，但都不能保证 100% 绕过验证码。

## 4. 三个采集入口

建采集任务（`POST /tasks`）时 `entry_type` 支持三种，配合 `provider=composer` 即走真实抓取：

| `entry_type` | `entry_value` | composer 内部调用 |
| --- | --- | --- |
| `keyword` | 搜索关键词（如 `телефон`） | `search_by_keyword(kw, page)` |
| `category` | 类目页 URL | `list_by_category(category_url, page)` |
| `seller` | 卖家 ID | `list_by_seller(seller_id, page)` |

三者共用同一套 cookie/代理/退避逻辑，区别只在拼的请求路径不同（见 `app/services/ozon_market/composer.py`）。`own_shop` 入口类型面向自建分支，不经 composer 抓取。

## 5. 反爬硬化设计

`OzonComposerProvider`（`app/services/ozon_market/composer.py`）：

- **请求前**：随机 UA（`_UA_POOL` 里挑一个）+ `_min_delay`~`_max_delay` 秒随机等待，降低请求特征的规律性。
- **cookie 注入**：构造时传入的 `cookie`（str 形式作为 `Cookie` 请求头，或 dict 形式走 `httpx` 的 `cookies=`）。
- **反爬信号识别**：响应状态码命中 `{429, 403, 307, 301, 302}`（限流/禁止/重定向到验证页）视为被拦截，按 `1s, 2s, 4s, ...` 指数退避重试，最多 `max_retries` 次（默认 4）。
- **耗尽后**：抛出 `CrawlerBlockedError`（`RuntimeError` 子类），错误信息里带「疑似反爬/cookie 失效，请在爬虫配置更新 cookie 或代理」的可操作提示。
- **采集任务不静默**：`run_collect_core`（`app/workers/collector.py`）捕获到 `CrawlerBlockedError` 后，把任务 `status` 置为 `failed`、`stats.error` 写入具体错误信息并落库，用户在任务列表/监控页能直接看到失败原因，而不是任务卡住或悄悄跑不出数据。

`server/tests/test_composer_hardening.py` 用 `httpx.MockTransport` 覆盖了 cookie 头注入、307 重试后成功、持续 403 抛 `CrawlerBlockedError` 三种场景。

## 6. 解析层独立（端点/结构变更只改 parser）

`app/services/ozon_market/parser.py::parse_search_widgets` 是唯一知道真实 Ozon 响应结构细节的地方，`composer.py` 只负责把请求发出去、拿到 JSON，解析全部委托给它。这样设计的原因：Ozon 前端结构（`widgetStates` 里的 key 命名、`mainState` 原子布局）会随其前端迭代变化，**未来结构变了只需要改 parser，不用动请求/重试逻辑**。

当前对齐的真实结构（浏览器实抓验证，详见 parser.py 顶部注释）：

- 商品挂在 `widgetStates` 里 key 含 `tileGrid` 或 `searchResults` 的条目下，value 是需要 `json.loads` 的 JSON 字符串，取其 `.items` 列表。
- 每个 item 的 `sku`、`action.link`（拼 `https://www.ozon.ru` 前缀得到商品链接）、`tileImage.items[].image.link`（图片列表）直接取值。
- 标题/价格/评分/评价数都藏在 `mainState`（原子列表）里，按 `type` 分派：`textDS` 里 `testInfo.automatizationId=="tile-name"` 的是标题；`priceV2` 里 `textStyle=="PRICE"` 的条目是当前价；`labelListV2` 的 text 项里第一个 0~5 小数是评分、第一个纯数字是评价数。
- 任何字段缺失均容错为 `None`/空列表，不抛异常——保证一条商品的字段缺失不影响整页解析。

`server/tests/fixtures/composer_search.json` 是一份真实抓包样本，`server/tests/test_parser_real.py` 以此为夹具驱动单测，保证解析逻辑与真实响应对齐。

## 7. Ozon Seller provider：real/mock 切换

真实抓取（composer 采集）与真实挂靠（Ozon Seller 写入）是两件独立的事——本任务只覆盖前者，后者见 M4/M5 文档。切换入口：

- 后台 `/settings/system`（admin）：`ozon_seller_provider` 设为 `mock`（默认）或 `real`。
- `run_publish`（`POST /listing/publish?sync=false` 入队）与 `run_publish_tick`（`/publish/tick?sync=false` 及 cron）会读该配置，读不到时回退环境变量 `OZON_SELLER_PROVIDER`。
- `sync=true` 的同步路径**恒用 `mock`**，不受此配置影响（详见 README「M4 已实现功能」）。
- `RealOzonSeller` 的挂靠/建品端点仍是占位实现（`create_follow_offer`/`create_product`/`get_product_status`），真实字段与状态映射需要 live 联调时校正，见第 9 节已知后置项。

## 8. `@live` 真实抓取测试跑法

`server/tests/test_live_crawler.py` 用 `@pytest.mark.live` 标记，`pyproject.toml` 里 `addopts = "-m 'not live'"` 让它在日常 `pytest` 全量跑里**默认被 deselect**，不会因为没有 cookie/网络而拖垮 CI。手动触发：

```bash
cd server
OZON_COOKIE='<从浏览器 DevTools 复制的完整 Cookie 值>' \
OZON_PROXY='<可选，代理地址>' \
.venv/bin/python -m pytest tests/test_live_crawler.py -m live -v
```

- 未设 `OZON_COOKIE` 时用例会 `pytest.skip`，不会误报失败。
- 断言：`search_by_keyword("телефон", 1)` 返回非空列表，且第一条有 `sku`/`title`。
- 若长期跑不出结果，先按第 2 节重新走一遍浏览器人工验证 + 换 cookie，再排查是否需要配代理。

仓库里另有 `server/tests/test_composer_live.py`，是更早期、更宽松的 live 探测（不要求非空、不需要 cookie），两者可以配合排障：先跑 `test_composer_live.py` 看端点是否可达，再跑 `test_live_crawler.py` 验证带 cookie 后能否解析出商品。

## 9. 已知 live 后置项

以下问题在 mock-first 阶段暂不处理，留待真实联调时解决：

- **cookie 池轮换**：当前只支持配置单条 cookie，长时间/高频采集容易被针对性限流；后续应支持多条 cookie 轮换 + 自动剔除失效 cookie。
- **`RealCategoryTree`**：`app/services/category_tree.py::CategoryTreeProvider` 的 `real` 实现（走 composer-api `categoryChildV3`）目前只是占位 import，未实现。
- **端点校正**：`RealOzonSeller` 的 `create_follow_offer`/`create_product`/`get_product_status` 请求地址与字段（`product/import`、`product/info` 等）是占位实现，真实上线前需按 Ozon Seller API 文档与实测响应校正。
- **多 API 实例限流需 Redis**：当前的间隔抖动/退避是单进程内的，多个 `worker` 实例并发跑真实采集时缺乏跨进程的统一限流，需要引入 Redis（如令牌桶）协调多实例的请求节奏，避免叠加请求量触发更严格的风控。
