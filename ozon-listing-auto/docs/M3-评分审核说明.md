# M3 使用说明：五维评分与审核台（打分 → 人工审核/自动采用）

本说明面向已完成 M1 采集、M2 货源匹配的用户，走一遍 M3 五维评分与审核台的完整操作流程，并说明如何切换到真实 LLM。

## 0. 前置说明：mock 优先，保持轻量

- 与 M2 一致，M3 的验收标准也是「`docker compose up`（mock embedder + mock LLM）一键跑通」。默认 `LLM_PROVIDER=mock`：`MockLLM` 是确定性桩（`translate` 恒等透传、`extract_json` 返回空字典），无需任何 key、无需网络，用于本地/CI 全链路验证。
- 只有需要真实译标题、真实抽属性以提升评分准确度时，才按第 5 节切换到真实 LLM。

## 1. 完整流程

```
采集(M1: 建任务→启动采集→商品入库)
  → 匹配(M2: /match/start→候选采集→CLIP 跨平台去重)
    → 评分(M3: /score/start→五维分+总分+tier)
      → 审核(M3: 审核台人工采用/拒绝 或 关审核开关按阈值自动采用)
```

## 2. 启动评分

在已有匹配候选（M2 `/candidates` 非空）的任务上：

```
POST /score/start?task_id=<id>&sync=true
```

- `sync=true`：请求内同步跑完（适合本地/mock 快速演示）。
- `sync=false`（默认）：经 ARQ 入队，由 `worker` 容器异步执行（`app.workers.scorer.run_score`），适合候选量大的场景（避免长请求超时）。

评分 worker 对每个商品：

1. 若 Ozon 商品主图向量（`embedding`）尚未算过，先用当前 `EMBEDDER`（mock/clip，同 M2）算好写回，避免重复计算。
2. 遍历该商品下的候选，逐个调用 `score_candidate()` 算五维分：
   - **图像 45%**：Ozon 主图向量与候选图向量的余弦相似度（× 100）
   - **标题 20%**：Ozon 标题经 LLM `translate()` 译中后，与候选标题做序列相似度
   - **属性 15%**：LLM `extract_json()` 从候选标题抽取结构化属性，与 Ozon 商品属性做键值命中率
   - **价格 5%**：命中价格区间给满分，否则按「有价/无价」给基础分
   - **供应商 15%**：综合候选的供应商复购率、信用等级（1688 `supplier_info`，来自 M2）等信号
   - 总分 = 五维加权和，权重默认 `{image:0.45, title:0.20, attr:0.15, price:0.05, supplier:0.15}`，调用方可传 `weights` 覆盖
   - 按阈值分级为 `tier`：`total >= tier_auto(默认85)` → `auto`；`total >= tier_review(默认70)` → `review`；否则 `rejected`
3. 分数与 tier 写回候选（`score_image`/`score_title`/`score_attr`/`score_price`/`score_supplier`/`score_total`/`tier`/`score_detail`），游标 (`score_cursor`) 记录到已处理的最后一个商品 id，暂停/失败后可从游标续跑。

查看进度：

```
GET /score/monitor?task_id=<id>
→ {"score_status": "running|done|paused|failed", "score_stats": {"products":..,"candidates_scored":..}}
```

暂停：`POST /score/pause?task_id=<id>`（下次 `/score/start` 会从游标续跑）。

## 3. 审核台

打分完成后，候选进入审核环节。任务上的 `review_config`（建任务/编辑任务时设置）决定走哪条路径：

```json
{"source_review_required": true, "source_score_min": 85}
```

- `source_review_required: true`（默认）：**需要人工审核**，候选停留在 `status="candidate"`，等待审核台处理。
- `source_review_required: false`：**免人工审核**，可调用 `/review/auto-adopt` 按 `source_score_min` 阈值批量自动采用。

### 3.1 人工审核（`source_review_required=true`）

打开审核台（前端 `ReviewBoard` 页面），输入任务 ID：

1. 点「拉取审核队列」→ `GET /review/queue?task_id=<id>`：按任务聚合返回「一个 Ozon 商品 ↔ 其下多个候选（按总分降序）」列表。
2. 页面**左侧**展示当前 Ozon 商品（主图/标题/SKU/价格），**右侧**逐张卡片展示该商品下的每个候选：平台标签（`ali1688`/`pinduoduo`）、候选标题、总分 + tier（`auto`=绿/`review`=金/`rejected`=红）、五维分明细（图/标题/属性/价/供应商）、候选价格。
3. 对候选点「采用」或「拒绝」→ `POST /review/{candidate_id}`（`decision=adopt|reject`），写入 `review_decisions` 留痕（含 `reviewer_id`/`note`/`created_at`）；候选状态相应更新为 `adopted`/`rejected`。
4. 同一商品下有多个候选时，逐个「采用」/「拒绝」即可实现「换候选」——例如先拒绝分数较低的候选，再采用另一个更合适的候选。
5. 「上一条」/「下一条」在商品间翻页复核。

### 3.2 关闭审核 · 自动采用（`source_review_required=false`）

审核台上有「关闭审核·自动采用」按钮，点击后二次确认（Popconfirm 弹窗提示「关闭人工审核将按阈值自动采用达标候选」），确认后调用：

```
POST /review/auto-adopt?task_id=<id>
→ {"auto_adopted": <n>}
```

服务端逻辑（`apply_auto_adopt`）：读取任务 `review_config`，若 `source_review_required=false`，则把该任务下 `status="candidate"` 且（若设了 `source_score_min`）`score_total >= source_score_min` 的候选批量置为 `status="auto_adopted"`，并写入 `review_decisions`（`decision="auto_adopt"`，`reviewer_id=null`，标记非人工操作但仍留痕可追溯）。未达阈值的候选保持 `candidate` 状态，不会被自动采用。

### 3.3 多人审核并发

多个审核员可能同时打开审核台处理同一批候选。`decide()` 内部通过 `review_lock(product_id)` 按商品加锁后再改状态、写决策，避免同一候选被重复决策。当前 `review_lock()` 是一个 no-op 上下文管理器（`app/services/review.py`），生产部署多审核员并发场景时，需将其替换为基于 `REDIS_URL` 的 Redis 分布式锁（如 `redis.lock.Lock`）以获得真正的互斥效果；单机/单审核员场景或测试环境下 no-op 实现不影响正确性。

## 4. 评分/审核相关字段速查

| 位置 | 字段 | 说明 |
| --- | --- | --- |
| `ozon_products` | `embedding` | 主图向量（pgvector，512 维），供图像分复用 |
| `supply_candidates` | `score_image`/`score_title`/`score_attr`/`score_price`/`score_supplier`/`score_total` | 五维分与总分 |
| `supply_candidates` | `tier` | `auto`/`review`/`rejected` |
| `supply_candidates` | `score_detail` | 打分明细 JSON（含译文等中间结果） |
| `collect_tasks` | `score_status`/`score_cursor`/`score_stats` | 评分任务状态/断点游标/统计 |
| `collect_tasks` | `review_config` | `{"source_review_required": bool, "source_score_min": float|null}` |
| `review_decisions` | `task_id`/`ozon_product_id`/`candidate_id`/`reviewer_id`/`decision`/`note`/`created_at` | 审核决策留痕（`adopt`/`reject`/`auto_adopt`） |

## 5. 切换到真实 LLM

mock 链路验证通过后，切换 `LLM_PROVIDER=openai` 即可让评分引擎的「译标题」「抽属性」调用真实模型（默认对接**通义千问 DashScope**，也可指向任意 OpenAI 兼容 Chat Completions 服务，如自建网关/其他厂商兼容接口）：

1. 编辑 `.env`：
   ```
   LLM_PROVIDER=openai
   LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1   # 或其他 OpenAI 兼容服务地址
   LLM_API_KEY=<真实 API key>
   LLM_MODEL=qwen-plus                                                # 或目标服务的模型名
   ```
2. 重新构建并启动 `worker` 服务：
   ```bash
   docker compose up -d --build worker
   ```
   `worker` 消费 `POST /score/start?...&sync=false`（默认）入队的异步任务（`app.workers.scorer.run_score`），启动时按 `settings.llm_provider` 选择 `MockLLM` 还是 `OpenAICompatLLM`。
   注意：与 M2 匹配的同步路径一致，`/score/start?sync=true` 出于本地/演示/测试目的**固定使用 `MockLLM` + `mock` embedder**（见 `app/api/score.py`），不受 `.env` 里 `LLM_PROVIDER`/`EMBEDDER` 影响；要验证真实 LLM 效果必须走 `sync=false` 的异步入队路径。
3. 切换后（`sync=false` 路径）`score_candidate()` 会真正调用 `OpenAICompatLLM`：`translate()` 请求模型把 Ozon 标题译成中文，`extract_json()` 请求模型从候选标题抽取结构化属性 JSON（内置 3 次重试 + 容错解析，模型返回带 ```json 代码围栏或夹杂说明文字也能解析出 JSON）。
4. 对应真实网络冒烟测试 `server/tests/test_llm_openai.py` 中标记 `@pytest.mark.live` 的用例默认跳过，需要显式执行 `pytest -m live` 才会真实请求模型接口（需要提前配置好可用的 `LLM_API_KEY`）。
5. 图像分若也要用真实 CLIP 语义向量（而非 mock 归一化向量），需同时按 [`docs/M2-货源匹配说明.md`](./M2-货源匹配说明.md) 第 5.2 节切换 `EMBEDDER=clip` + `INSTALL_ML=true`（重新构建 `worker` 镜像装 torch/cn_clip）；LLM 与 embedder 两个开关相互独立，可分别按需切换。

## 常见问题

- **`/score/start` 返回 `queued` 后候选分数迟迟不出现？** 确认 `worker` 容器已启动且能连上 Redis/Postgres（`docker compose logs worker`），用 `GET /score/monitor` 看 `score_status` 是否推进。
- **候选一直没有 `tier`？** 说明还没跑过 `/score/start`；候选刚从 M2 匹配产出时 `score_total`/`tier` 均为空，只有评分完成后才会写入。
- **`/review/auto-adopt` 返回 `auto_adopted: 0`？** 检查任务 `review_config.source_review_required` 是否确实为 `false`（为 `true` 时接口直接返回 0，不做任何操作），以及候选 `score_total` 是否达到 `source_score_min`。
- **设置了 `LLM_PROVIDER=openai` 但标题分看起来没变化？** 确认改的是 `worker` 服务的环境变量并已重启该容器，且请求走的是 `sync=false`（异步入队）路径——`sync=true` 的同步演示路径固定使用 `MockLLM`；同时确认 `LLM_API_KEY`/`LLM_BASE_URL`/`LLM_MODEL` 配置正确，接口不可用时 `OpenAICompatLLM.chat()` 重试 3 次后会抛异常，导致该商品评分失败（`score_status=failed`）。
