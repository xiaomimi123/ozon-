# M1 使用说明：采集 → 筛选 → 商品列表 → 选模式

本说明面向已按 README 完成 `docker compose up`（或本机 `uvicorn` + `npm run dev`）启动的用户，走一遍 M1 的完整操作流程。

## 1. 登录

浏览器打开前端地址：
- Docker Compose 方式：http://localhost:8080
- 本机开发方式：`npm run dev` 打印的地址（默认 http://localhost:5173）

使用默认管理员账号登录：

```
用户名：admin
密码：admin123
```

（该账号由后端启动钩子幂等创建，若已通过 `ADMIN_USER` / `ADMIN_PASSWORD` 覆盖过，则用你设置的账号密码。）

## 2. 新建采集任务

进入「任务中心」页面，点新建任务，填写：

| 字段 | 说明 |
| --- | --- |
| 名称 | 任意，便于区分任务 |
| 跟卖 / 自建（`listing_mode`） | `follow`（跟卖，默认）或 `create`（自建）——M1 只存储该选择，具体消费逻辑在后续里程碑（M3/M4） |
| 入口类型（`entry_type`） | `keyword`（关键词）/ `category`（类目）/ `seller`（店铺）/ `own_shop`（自有店铺）之一 |
| 入口值（`entry_value`） | 对应入口类型的具体值，如关键词文本、类目/店铺 URL |
| 数据源（`provider`） | **建议先选 `mock`**，无需任何外部配置即可跑通全流程；后续可切 `composer` 接真实爬虫（见第 5 节） |

提交后任务状态为 `pending`。

## 3. 启动采集

在任务列表中对目标任务点「启动」（对应 `POST /collect/start?task_id=`），任务状态变为 `running`。

- 采集过程通过 WebSocket（`/ws/progress`）实时推送进度：当前页、已采集/去重计数。
- 采集会自动做**跨页去重**（同 `(task_id, sku)` 唯一）与**变体归集**（按 `parent_sku` 归并）。
- 需要暂停时点「暂停」（对应 `POST /collect/pause?task_id=`），任务状态变为 `paused`；再次点「启动」会从上次的游标（`cursor`）**断点续传**，不会重复采集或丢数据。
- `provider=mock` 下采集速度很快（无外部网络请求），适合本地演示与调试。

## 4. 商品列表：六维筛选

采集有数据后，进入「商品列表 / 筛选」页面，选择对应任务，即可看到已入库商品（图片、标题、价格、月销、评分、跟卖数……）。

六个可空维度（不填即不限制该维度）：

1. 月销量（`sales_min` / `sales_max`）
2. 评分下限（`rating_min`）
3. 退货率上限（`return_rate_max`）
4. 重量区间（`weight_min` / `weight_max`）
5. 上架时间下限（`listed_after`）
6. 跟卖/关注数区间（`follow_min` / `follow_max`）

另支持标题关键词模糊搜索（`keyword`）。

调整任意条件后重新查询即为「可改重筛」——所有条件均可清空、修改，随时重新提交查看新结果，支持分页（`page` / `page_size`，均从 1 起、`page_size` 上限 200）。

## 5. 切换到真实采集：provider=composer

`mock` 用于本地验证链路；要接真实 Ozon 数据，需要将任务的 `provider` 改为 `composer`：

1. **配置 cookie / 代理**：以 `admin` 身份调用配置中心接口 `PUT /settings/{category}`（写入的值会加密存储，读取时脱敏返回），填入可用的 cookie、代理地址等凭据。当前 `OzonComposerProvider` 已内置随机 UA、请求间隔抖动、429/403 指数退避，但反爬策略会随时间变化，实际可用性依赖你配置的 cookie/代理是否有效，需要按需调优。
2. 新建（或编辑）任务时把 `provider` 选为 `composer`，其余字段（`entry_type`/`entry_value` 等）填法与 `mock` 一致——上层（采集 worker、去重、筛选、商品列表）对 provider 切换无感。
3. 启动采集，观察 WS 进度与商品列表是否有真实数据入库；若被拦截（403/429 重试耗尽会抛异常），回配置中心调整 cookie/代理后重试。
4. 对应的真实网络冒烟测试 `server/tests/test_composer_live.py` 默认跳过（标记 `@pytest.mark.live`），需要显式执行 `pytest -m live` 才会真实发起网络请求，日常回归不会因外部网络波动失败。

## 常见问题

- **启动采集后商品列表没数据？** 确认任务状态是否变为 `running`/`done`，以及商品列表页选中的任务是否正确；`mock` provider 下应该秒级出结果。
- **composer 采集报「composer-api 反爬拦截」？** 说明当前 cookie/代理已失效或触发风控，需要更新配置中心里的凭据。
- **想清空筛选条件重新看全部商品？** 把六个维度全部清空、关键词留空，重新查询即可。
